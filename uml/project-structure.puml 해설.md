# SafeCrowd UML 설계 해설 - project-structure.puml

대상 파일: `uml/project-structure.puml`

## 문서 목적
이 문서는 전체 계층 구조 UML에 등장하는 각 요소가 무엇이고, 왜 필요한지, 구현 시 무엇을 조심해야 하는지, 이후 어떤 방향으로 확장할 수 있는지를 정리한 문서이다.

이 그림은 SafeCrowd 전체를 가장 상위 수준에서 설명하는 구조도이므로, 세부 구현보다 **계층 경계와 책임 분리**를 읽는 데 초점을 둔다.

---

## `safecrowd_app (Qt App)`
- 개요: Qt 기반 실행 파일과 사용자 인터페이스 진입점이다.
- 목적: 시나리오 편집, 실행 제어, 결과 표시 같은 사용자 상호작용을 담당한다.
- 유의사항: 도메인 계산이나 ECS 내부 규칙을 직접 넣지 말고, 화면과 도메인 호출 연결에 집중해야 한다.
- 후속 개선 사항: viewport bridge가 실제로 필요해지면 UI 스레드와 엔진 실행 경계를 더 명확히 문서화할 수 있다.

## `ScenarioDefinition`
- 개요: 공간 구조, 인원 배치, 환경 조건, 운영 계획을 묶는 시나리오 정의 모델이다.
- 목적: UI에서 만든 입력을 단일 실행과 배치 비교 모두에서 공통 계약으로 사용하기 위해 필요하다.
- 유의사항: 엔진용 내부 파라미터와 사용자 입력 모델을 그대로 섞지 말고, 상위 개념 중심의 정의를 유지하는 편이 좋다.
- 후속 개선 사항: 저장 포맷 버전 관리, preset 참조, 변경 이력 추적을 나중에 붙일 수 있다.

## `ExecutionConfig`
- 개요: 제한시간, 샘플링 주기, seed, 반복 횟수 같은 실행 계약이다.
- 목적: 같은 시나리오를 어떤 조건으로 실행했는지 재현 가능하게 남기기 위해 필요하다.
- 유의사항: 시나리오 내용과 실험 조건을 구분해야 비교 결과가 덜 혼란스럽다.
- 후속 개선 사항: 멀티런 통계 정책, 시간 제한 정책, 실패 처리 정책을 더 세분화할 수 있다.

## `ScenarioCalibrationService`
- 개요: UI 입력을 엔진과 도메인이 쓰는 내부 파라미터로 변환하는 보정 계층이다.
- 목적: 사용자 친화적 입력 모델과 저수준 시뮬레이션 파라미터 사이를 분리하기 위해 필요하다.
- 유의사항: 보정 로직이 엔진 레이어로 새지 않게 하고, 도메인 규칙 안에 두는 것이 맞다.
- 후속 개선 사항: 공간 유형별 preset curve, 연구 모드용 세부 파라미터 노출, 보정 결과 검증 리포트를 추가할 수 있다.

## `SimulationSession`
- 개요: 단일 시뮬레이션 한 번의 수명주기를 오케스트레이션하는 도메인 계층 객체다.
- 목적: 하나의 실행이 엔진 초기화, 실행, 수집, 종료를 일관된 절차로 따르도록 만들기 위해 필요하다.
- 유의사항: 세션은 도메인 흐름을 조정하는 역할이어야지, 엔진 내부 storage 구조를 직접 다루는 객체가 되면 안 된다.
- 후속 개선 사항: 단계별 hook, 상태 전이 로그, 실행 중단/재개 정책을 추가할 수 있다.

## `ScenarioBatchRunner`
- 개요: 기준 시나리오와 대안 시나리오를 반복 실행하고 비교하는 배치 실행 조정자다.
- 목적: SafeCrowd의 핵심 가치인 시나리오 비교를 단일 실행 위에 얹기 위해 필요하다.
- 유의사항: 병렬화나 실행 격리는 나중에 붙일 수 있지만, 지금도 단일 실행 계약은 명확해야 한다.
- 후속 개선 사항: 병렬 실행, 작업 큐, 실패 복구, 결과 캐시를 나중에 확장할 수 있다.

## `ResultAggregator`
- 개요: 여러 실행 결과를 비교 가능한 요약 지표와 히트맵, delta 뷰로 집계하는 요소다.
- 목적: 단일 실행 로그를 바로 보여주는 대신 의사결정용 비교 결과를 만들기 위해 필요하다.
- 유의사항: 평균값만 모으는 데 그치지 말고 최악값, 퍼센타일, 기준 대비 변화율을 함께 고려해야 한다.
- 후속 개선 사항: 정규화 지표, 반복 실행 통계 요약, 위험 구역 전이 분석을 추가할 수 있다.

## `EngineRuntime`
- 개요: 엔진의 실행 루프와 수명주기를 관리하는 API 진입점이다.
- 목적: 도메인이 엔진 세부 구현을 몰라도 initialize/play/stop 같은 제어를 할 수 있게 한다.
- 유의사항: 런타임은 orchestration 역할에 집중하고, crowd 전용 규칙을 넣지 않는 것이 중요하다.
- 후속 개선 사항: pause reason, 상태 진단, 디버그용 실행 제어 기능을 확장할 수 있다.

## `EngineWorld`
- 개요: query, resources, commands를 묶은 공유 월드 파사드다.
- 목적: 도메인과 runtime이 raw ECS storage 대신 공통 접근면을 통해 월드에 접근하도록 만들기 위해 필요하다.
- 유의사항: 읽기와 쓰기 권한이 한 객체에 모여 있으므로, 사용 규칙이 흐려지지 않게 command 지연 반영 원칙을 유지해야 한다.
- 후속 개선 사항: 필요해지면 `ReadOnlyWorld`와 `CommandContext` 같은 분리된 접근면으로 확장할 수 있다.

## `EngineSystem`
- 개요: 엔진에서 등록되고 스케줄링되는 범용 시스템 계약이다.
- 목적: 도메인 시스템과 엔진 공통 시스템이 같은 실행 규칙을 따르도록 만들기 위해 필요하다.
- 유의사항: 시스템 인터페이스는 전역 싱글턴 의존 대신 명시적 입력을 받는 구조를 유지하는 편이 좋다.
- 후속 개선 사항: 시작/종료 hook, 조건부 실행, 프로파일링 메타데이터를 추가할 수 있다.

## `Runtime Core`
- 개요: `FrameClock`, `SystemScheduler`, `CommandBuffer`, `ResourceStore`, `DeterministicRng`를 묶은 실행 코어 영역이다.
- 목적: 시간 관리, 시스템 실행 순서, 지연 변경, 리소스 저장, 재현성을 엔진 공통 기능으로 제공한다.
- 유의사항: 초기 단계에서는 기능을 넓히기보다 fixed-step과 deferred mutation 규칙을 먼저 단단히 잡는 편이 좋다.
- 후속 개선 사항: 이벤트 기반 스케줄링, 스냅샷, 실행 계측, 병렬 phase 실행으로 확장할 수 있다.

## `ECS Core`
- 개요: `EntityRegistry`, `ComponentRegistry`, `PackedComponentStorage` 중심의 밀집 저장 영역이다.
- 목적: 엔티티와 컴포넌트를 효율적으로 저장하고 query 기반 순회를 가능하게 하기 위해 필요하다.
- 유의사항: 엔진 재사용성을 위해 SafeCrowd 도메인 타입이나 위험 계산 로직이 이 계층으로 들어오면 안 된다.
- 후속 개선 사항: query cache, archetype 유사 구조, 공간 인덱스와의 결합 가능성을 나중에 검토할 수 있다.
