# Sprint 시연 계획

## 목적
이 문서는 교수님 앞에서 각 Sprint 종료 시점에 어떤 수준까지 제품을 시연해야 하는지 간결하게 정리한다. 상세 요구와 전체 범위는 `Product Backlog.md`를 따르고, 이 문서는 시연 기준만 요약한다.

## Sprint 1 시연
목표: 도면을 불러오고, 기본 시나리오를 만들어, 실제로 한 번 실행되는 최소 제품 흐름을 보여 준다.

- 도면 import, 구조 검토, 차단 이슈 확인
- 레이아웃 수동 보정과 실행 가능 상태 갱신
- baseline 시나리오 생성과 필수 입력 편집
- 운영 이벤트 기본 설정
- run / pause / stop 실행 제어
- 실시간 상태 표시와 기본 viewport 시연

한 줄 요약: `도면 불러오기 -> 검토/보정 -> 시나리오 작성 -> 실행`이 끊기지 않고 동작해야 한다.

## Sprint 2 시연
목표: Sprint 1 흐름 위에 결과 분석과 비교 기능이 붙어 제품이 실제 의사결정 도구처럼 보이게 한다.

- 반복 실행과 run / variation 구분
- run 결과 요약과 variation summary 확인
- heatmap 및 위험 레이어 표시
- baseline 대비 alternative 비교
- 변경 변수와 주요 위험 지표 차이 확인

한 줄 요약: `실행된다`에서 끝나는 것이 아니라 `결과를 읽고 비교할 수 있다`까지 보여 줘야 한다.

## Sprint 3 시연
목표: 제품을 마무리해 저장, 내보내기, 추천 흐름까지 포함한 완성형 데모를 보여 준다.

- 시나리오 템플릿 기반 빠른 시작
- 추가 위험 축 반영 결과 확인
- 프로젝트 저장 및 다시 열기
- 결과 export
- 운영 대안 추천과 추천안 검토

한 줄 요약: `작성 -> 실행 -> 분석 -> 저장/공유 -> 추천 검토`까지 이어지는 완성형 흐름을 보여 준다.

## 시연 관점의 우선순위
- Sprint 1: 기본 실행 흐름이 반드시 끊기지 않아야 한다.
- Sprint 2: 결과 비교와 분석 근거가 눈에 보여야 한다.
- Sprint 3: 완성도와 발표용 흐름이 정리되어 있어야 한다.
