# SafeCrowd 시뮬레이션: 위험 정의

## 개요
본 문서는 SafeCrowd의 제품 요구 관점에서 위험을 어떻게 보여주고 비교할지를 정의하는 기준 문서다. Pathfinder 2026.1 공식 문서를 참고해 사용자 입력과 결과 비교 중심으로 정리했고, 세부 물리 모델과 구현 참고는 [고급 위험 모델.md](고급%20위험%20모델.md)로 분리한다.

정리의 기준은 Pathfinder 2026.1 공식 문서가 실제 제품 기능을 어떤 단위로 노출하는지에 맞춘 것이다. Pathfinder는 위험을 별도의 대형 taxonomy로 정의하기보다, `출구 선택`, `행동 전환`, `측정 구역`, `혼잡/밀도 시각화`, `시야`, `FED`, `근접도` 같은 입력과 결과 단위로 나누어 다룬다.

이 문서의 목적은 다음과 같다.

- SafeCrowd의 위험 정의를 사용자 입력과 결과 비교 중심으로 단순화한다.
- 현재 제품 범위와 이후 확장 범위를 분리한다.
- 구현 세부사항과 제품 요구를 한 문서에서 강하게 결합하지 않는다.

## Pathfinder 비교 기준

Pathfinder 2026.1 공식 문서를 다시 확인한 결과, 현재 제품 기능의 중심은 다음과 같이 정리할 수 있다.

- 출구 선택은 `Current Room Travel Time`, `Queue Time`, `Global Travel Time` 같은 비용 요소 조합으로 설명된다. [Door Choice Tab](https://www.thunderheadeng.com/docs/2026-1/pathfinder/profiles/door-choice/)
- 행동은 `Wait`, `Goto`, 목표 변경, 행동 시퀀스 추가 같은 단위로 다룬다. [Creating a New Behavior](https://www.thunderheadeng.com/docs/2026-1/pathfinder/behaviors/creating-new/)
- 운영 변화는 trigger 인지, 인식, 결정, 사용 단계로 나누어 다룬다. [Occupant Response to a Trigger](https://www.thunderheadeng.com/docs/2026-1/pathfinder/triggers/response/)
- 결과는 측정 구역과 열지도 중심으로 제공된다. [Measurement Regions](https://www.thunderheadeng.com/docs/2026-1/pathfinder/output/measurement-regions/), [Occupant Contours / Heat Maps](https://www.thunderheadeng.com/docs/2026-1/results/viewing-pathfinder-output/occupant-contours/)
- 시야와 연기 노출은 별도 계산 결과로 다뤄진다. [Calculating Occupant Visibility through Smoke](https://www.thunderheadeng.com/docs/2026-1/results/viewing-pathfinder-output/visibility/), [Calculating Fractional Effective Dose (FED)](https://www.thunderheadeng.com/docs/2026-1/results/viewing-pathfinder-output/fed/)
- 대인 노출은 근접도 분석으로 다뤄진다. [Occupant Proximity Analysis](https://www.thunderheadeng.com/docs/2026-1/results/viewing-pathfinder-output/occupant-proximity/)

반대로 Pathfinder 공식 문서에서 직접 1급 기능으로 강하게 드러나지 않는 항목은 다음과 같다.

- 압사와 낙상의 독립 사고 모델
- 역방향 흐름의 `OrderParameter`, `LaneCount`
- 교차 흐름의 `StripeCoherence`
- ECS 컴포넌트와 시스템 이름 수준의 설계
- 운영 대안 추천의 상세 규칙과 최적화 로직

따라서 SafeCrowd의 제품 문서도 우선은 Pathfinder와 비슷하게 `입력`, `행동`, `측정`, `결과 비교` 중심으로 정리하고, 세부 위험 물리 모델은 후속 문서로 분리하는 편이 적절하다.

## 정리 원칙

- 제품 문서는 사용자가 설정하는 값과 비교에 쓰는 결과 지표를 먼저 정의한다.
- 위험 개념은 운영 의사결정에 직접 연결되는 수준까지만 노출한다.
- 엔진 구현 세부사항, ECS 명세, 저수준 파라미터 이름은 별도 설계 문서로 분리한다.
- 의학적 판정처럼 과도하게 강한 용어는 초기 버전의 핵심 모델에서 피한다.
- Pathfinder에서 이미 검증된 입력/결과 단위를 우선 차용하되, SafeCrowd는 그 위에 의사결정 지원 레이어를 얹는다.

## 적용 단계와 문서 역할

- 이 문서는 SafeCrowd의 제품 요구 관점에서 위험을 어떻게 보여주고 비교할지를 정리하는 기준 문서다.
- 세부 물리 모델, ECS 컴포넌트, 저수준 파라미터는 [고급 위험 모델.md](고급%20위험%20모델.md)에 두되, 그 문서 역시 `Pathfinder 정합 확장`과 `연구 후보`를 구분한 참고 문서로 취급한다.
- 운영 대안 추천 기능 자체는 제품 범위에서 제외하지 않는다. 다만 추천 엔진의 상세 규칙과 최적화 로직은 별도 문서로 분리한다.

### 단계 구분 원칙
- `MVP`
  - 병목, 정체, 탈출 지연
  - 운영 갈등
  - 기본 heatmap, 출구 처리량, 대피 시간, 구역 밀도/정체 시간
- `1차 확장`
  - 근접도 및 압박 전조의 정량화
  - 시야 제한, 공간 친숙도, 유도 신호에 따른 길찾기 저하
  - 그룹 분포, queue/target 연계 결과, generic results viewer 고도화
- `중기 확장`
  - FED/FDS 연동
  - 추천 규칙의 정교화와 자동 탐색 로직
- `연구 후보`
  - 압사, 낙상, 역방향/교차류의 독립 고급 모델
  - 연구 데이터 기반 정밀 판정 지표

- 즉, 이 문서의 네 위험 축은 제품 개념 모델로 유지하되, 각 축의 구현 깊이는 같은 단계로 올라가지 않는다.

## 1. SafeCrowd 기본 위험 체계

초기 제품 문서에서는 위험을 아래 네 축으로 정의한다.

### 1.1. 병목, 정체, 탈출 지연

### 정의
출구, 계단, 코너, 좁은 통로, 대기열 집중 구간에서 유입량이 처리 용량을 넘어서며 이동 속도가 떨어지고 대기 시간이 늘어나 전체 대피 시간이 증가하는 현상이다.

### 사용자가 직접 설정하는 주요 입력
- 공간 구조와 유효 폭
- 출구 개방 여부
- 인원 수와 초기 배치
- 순차 퇴장 여부
- 일방통행, 접근 제한, 우회 동선 같은 운영 규칙

### 핵심 결과 지표
- 구역별 밀도
- 구역별 혼잡 지속시간
- 출구 또는 연결부 처리량
- 대기 시간
- 총 대피 시간
- 90/95 퍼센타일 대피 시간
- 제한시간 내 미대피 인원

### Pathfinder 비교
이 범주는 Pathfinder의 `Door Choice`, `Measurement Regions`, `Occupant Contours / Heat Maps`와 가장 직접적으로 대응된다.

### 1.2. 시야 제한 및 길찾기 저하

### 정의
연기, 조명 저하, 구조 복잡성, 공간 비친숙성 때문에 출구 인지와 경로 유지가 어려워지고, 이동 속도와 경로 선택 성능이 저하되는 현상이다.

### 사용자가 직접 설정하는 주요 입력
- 시야 상태 또는 연기 수준
- 공간 친숙도
- 표지 가시성
- 유도 인력 또는 안내 신호
- 사용 가능한 출구 집합

### 핵심 결과 지표
- 이동 속도 저하
- 경로 우회 정도
- 출구 선택 변화
- 가시성 값
- 연기 노출 또는 FED 유사 지표

### Pathfinder 비교
이 범주는 Pathfinder Results의 `Visibility`와 `FED` 기능, 그리고 Pathfinder의 행동/출구 선택 설정과 대응된다.

기본 `시야/친숙도/유도 신호` 입력과 결과 비교는 `1차 확장` 범위로 보고, `FED/FDS` 같은 환경 연동은 `중기 확장`으로 둔다.

### 1.3. 근접도 및 압박 전조

### 정의
고밀도 구간에서 대인 간격이 지속적으로 줄어들고 벽면이나 장애물 주변에 군중이 압축되며, 위험한 압박 상태로 이어질 수 있는 전조가 나타나는 현상이다.

초기 버전에서는 이 범주를 의학적 `압사` 판정으로 다루지 않는다. 대신 운영 의사결정에 필요한 `고밀도 체류`, `근접 노출`, `압박 hotspot` 수준으로 제한한다.

### 사용자가 직접 설정하는 주요 입력
- 인원 밀집도
- 병목 폭과 병목 위치
- 장애물과 펜스 배치
- 출구 운영 방식
- 유입 분산 여부

### 핵심 결과 지표
- 대인 최소 거리
- 근접 노출시간
- 고밀도 지속시간
- 벽면 또는 장애물 인접 hotspot
- 혼잡 hotspot의 전이 여부

### Pathfinder 비교
이 범주는 Pathfinder의 `Interpersonal Distance`, `Occupant Proximity Analysis`, `Density` 계열 시각화와 가까운 수준까지를 우선 목표로 삼는다. `Compression asphyxia` 같은 직접 사고 판정은 후속 확장으로 미룬다.

MVP에서는 `고밀도 체류`, `압박 hotspot`, `혼잡 전이` 위주로 시작하고, `대인 최소 거리`, `근접 노출시간` 같은 정량화는 `1차 확장`으로 올리는 편이 현재 계획과 더 잘 맞는다.

### 1.4. 운영 갈등

### 정의
서로 다른 이동 방향, 교차 동선, 동시 유입, 잘못된 운영 규칙 때문에 국소적인 충돌 회피와 지연이 반복되고, 전체 흐름이 불안정해지는 현상이다.

초기 버전에서는 역방향 흐름과 교차 흐름을 별개의 고급 물리 모델로 완전히 분리하지 않고, 운영 갈등과 혼잡 심화의 하위 유형으로 묶는다.

### 사용자가 직접 설정하는 주요 입력
- 일방통행 여부
- 교차 동선 허용 여부
- 순차 퇴장과 release timing
- 구역 접근 제한
- 유도 인력과 통제 구역

### 핵심 결과 지표
- 갈등 hotspot
- 교차 지점 지연
- 방향별 처리량 불균형
- 운영 변경 전후의 혼잡 분산 효과

### Pathfinder 비교
이 범주는 Pathfinder의 trigger, behavior, queue, target 조합으로 모델링되는 운영 변화와 가장 가깝다. `LaneCount`, `OrderParameter`, `StripeCoherence`는 MVP의 필수 공개 지표로 두지 않는다.

## 2. 사용자 입력 계약

위험 문서는 사용자가 실제로 설정하는 입력 단위를 먼저 정의해야 한다. 기본 입력 계약은 다음 다섯 묶음으로 유지한다.

- `FacilityLayout`: 방, 복도, 계단, 출구, 장애물, 펜스, 통제 가능 구역(`ControlZone`)
- `PopulationSpec`: 초기 배치, 동적 유입, `PopulationProfile`(이동 약자 비율, 기본 이동 성향, 필요 시 그룹 분포)
- `EnvironmentState`: 시야, 친숙도, 조명, 표지 가시성, 유도 인력, 필요 시 연기/FED 연동
- `ControlPlan`: 출구 개방/폐쇄, 일방통행, 접근 제한, 순차 퇴장, 임시 장애물, 유입 통제
- `ExecutionConfig`: 제한시간, 샘플링 주기, 반복 실행 수, 난수 seed

제품 문서에서 `PopulationProfile`은 독립 top-level 입력 계약이 아니라 `PopulationSpec`의 하위 구성요소로 취급한다.

초기 제품 문서에서는 `DesiredSpeed`, `FollowingWeight`, `CompressionExposure`, `OrderParameter` 같은 내부 파라미터를 직접 계약의 일부로 두지 않는다.

단계 적용 기준은 다음과 같다.

- `MVP`: `FacilityLayout`, 기본 `PopulationSpec`, `ControlPlan`, `ExecutionConfig`
- `1차 확장`: 그룹 분포, 시야/친숙도/유도 신호 입력
- `중기 확장`: 연기/FED/FDS 연동

## 3. 결과 지표

### 3.1. 실시간 표시

- 밀도 heatmap
- 혼잡 heatmap
- 출구와 연결부의 queue 상태
- 시야 또는 연기 상태
- 근접도 또는 고밀도 hotspot
- 운영 이벤트 상태 변화

### 3.2. 시나리오 종료 후 비교 지표

- 총 대피 시간
- 50/90/95 퍼센타일 대피 시간
- 제한시간 내 미대피 인원
- 출구별 처리량
- 구역별 최대 밀도와 고밀도 지속시간
- 구역별 평균 속도와 정체 시간
- 근접 노출시간 요약
- 시야 또는 FED 유사 노출 요약
- 기준 시나리오 대비 변화율

이 중 `총 대피 시간`, `미대피 인원`, `출구별 처리량`, `구역별 밀도/정체`는 `MVP`, `근접 노출시간`과 기본 시야 저하 요약은 `1차 확장`, `FED` 연동 요약은 `중기 확장`으로 본다.

### 3.3. 초기 버전에서 제외하는 항목

- 의학적 사망 판정
- 낙상과 연쇄 전도의 독립 사고 모델
- 역방향 흐름용 lane stability 지표
- 교차 흐름용 stripe 지표
- 추천 엔진의 상세 규칙과 최적화 로직

이 항목들은 제품 가치가 없어서가 아니라, 초기 위험 문서를 불필요하게 복잡하게 만들기 때문에 별도 후속 문서로 분리한다.

## 4. backlog 및 설계 반영 기준

이 문서는 backlog와 UML에 그대로 복사하는 문장이 아니라, 제품 기준 용어를 정리하는 상위 입력으로 사용한다. 현재 문서 기준의 권장 매핑은 다음과 같다.

| 축/계약 | backlog 반영 위치 | 설계 반영 위치 | 단계 메모 |
| --- | --- | --- | --- |
| 병목, 정체, 탈출 지연 | `E4`, `E5` | 결과 아티팩트, connector/flow 측정 | `MVP` |
| 운영 갈등 | `E2`, `E4`, `E5` | behavior/trigger/control model | `MVP` |
| 근접도 및 압박 전조 | `E4`, `E5` | proximity, hotspot, measurement 구조 | `MVP hotspot`, `1차 확장 정량화` |
| 시야 제한 및 길찾기 저하 | `E2`, `E4`, `E5` | environment state, route choice, guidance | `1차 확장`, `FED/FDS는 중기` |
| 사용자 입력 계약 5종 | `E1`, `E2`, `E3` | `domain-scenario-model`의 핵심 입력 | 단계별 확장 |
| 결과 지표와 비교 방식 | `E5`, `E7` | `domain-result-artifacts`, results workflow | `MVP`부터 |
| 운영 대안 추천 | `E6` | recommendation workflow | 기능은 유지, 상세 규칙은 별도 문서 |

특히 현재 backlog에서 `압박 위험`, `낙상`, `역방향/교차 위험`, `추천`은 이미 스토리로 약속돼 있으므로, 이 문서를 기준으로 정리할 때는 "기능 제외"가 아니라 "제품 문서와 구현 문서의 역할 분리"로 해석하는 것이 안전하다. 다만 독립 연구 모델은 별도 설계 합의 전까지 현재 구현 계획의 기본 전제로 올리지 않는다.

## 5. 문서 분리 원칙

현재 문서 체계는 아래처럼 역할을 고정한다.

- `위험 정의.md`: 제품이 사용자에게 어떤 위험을 보여주고 어떤 결과를 비교하는지 정의하는 정본
- `고급 위험 모델.md`: Pathfinder 정합 확장과 연구 후보를 함께 정리한 참고 문서
- 별도 구현 문서: 내부 파라미터, ECS 컴포넌트, 시스템 분해, 계산 주기
- 별도 분석 문서: 압사, 낙상, 역방향/교차류 정밀 지표, 연구 모드 확장
- 별도 추천 문서: 위험 패턴과 운영 대안 매핑 규칙

즉, 위험 정의 기준 문서는 `사용자 개념과 결과 비교`를 담당하고, 구현과 연구 세부사항은 뒤로 분리하는 것이 맞다.

## 6. 후속 작업 제안

- [고급 위험 모델.md](고급%20위험%20모델.md)의 내용을 `Pathfinder 정합 확장`과 `연구 후보`가 섞이지 않게 유지한다.
- 압사, 낙상, 역방향 흐름, 교차 흐름의 세부 모델은 실제 구현 전에 별도 설계 문서로 한 번 더 분리한다.
- ECS 컴포넌트와 시스템 명세는 `architecture/` 또는 별도 구현 설계 문서로 이동한다.
- 추천 규칙은 제품 MVP 범위가 확정된 뒤 별도 문서에서 다룬다.
