# SafeCrowd 시뮬레이션: 고급 위험 모델

## 문서 역할 메모
- 제품 요구와 사용자 노출 위험 체계는 [위험 정의.md](위험%20정의.md)를 우선 기준으로 삼는다.
- 본 문서는 그보다 아래 레벨의 고급 확장 참고 문서다.
- 이 문서에 적힌 항목이 모두 현재 구현 약속이나 backlog 확정 범위를 뜻하지는 않는다.
- 내용은 `Pathfinder 정합 확장`과 `연구 후보`로 나누어 관리한다.

## 1. 단순화 기준
- Pathfinder 2026.1 공식 문서가 직접 입력, 행동, 출력, 검증 시나리오로 드러내는 항목만 `Pathfinder 정합 확장`으로 둔다.
- Pathfinder 공식 문서에 1급 기능으로 드러나지 않고, 별도 연구 논문과 독자 모델링이 필요한 항목은 `연구 후보`로 둔다.
- UML, issue, 구현 계획에는 우선 `Pathfinder 정합 확장`만 직접 연결한다.
- `연구 후보`는 별도 설계 문서와 검증 근거가 생기기 전까지 구현 계획의 기본 전제로 올리지 않는다.

## 2. Pathfinder 정합 확장

### 2.1. 경로 선택과 운영 반응
- Pathfinder는 출구 선택을 `Current Room Travel Time`, `Queue Time`, `Global Travel Time` 조합으로 설명한다.
  - 관련 문서: [Door Choice Tab](https://www.thunderheadeng.com/docs/2026-1/pathfinder/profiles/door-choice/)
- Trigger 반응도 단순 on/off가 아니라 `filtering -> awareness -> decision scheduling -> decision -> use` 단계로 설명한다.
  - 관련 문서: [Occupant Response to a Trigger](https://www.thunderheadeng.com/docs/2026-1/pathfinder/triggers/response/)
- Behavior, Group, Queue, Target은 독립 물리 위험 모델이 아니라 운영 시나리오를 구성하는 제어 단위로 제공된다.
  - 관련 문서: [Behaviors](https://www.thunderheadeng.com/docs/2026-1/pathfinder/behaviors/), [Movement Groups](https://www.thunderheadeng.com/docs/2026-1/pathfinder/groups/), [Queues](https://www.thunderheadeng.com/docs/2026-1/pathfinder/queues/), [Targets](https://www.thunderheadeng.com/docs/2026-1/pathfinder/targets/)
- SafeCrowd는 여기서 `경로 비용`, `행동 전환`, `운영 이벤트`, `태그 기반 제어`를 우선 차용하면 된다.

### 2.2. 측정 구역과 결과 아티팩트
- Pathfinder는 위험을 별도 사고 taxonomy보다 측정과 결과 아티팩트로 드러낸다.
  - 관련 문서: [Measurement Regions](https://www.thunderheadeng.com/docs/2026-1/pathfinder/output/measurement-regions/), [Summary Report](https://www.thunderheadeng.com/docs/2026-1/pathfinder/output/summary/), [Door History](https://www.thunderheadeng.com/docs/2026-1/pathfinder/output/doors/), [Room History](https://www.thunderheadeng.com/docs/2026-1/pathfinder/output/rooms/)
- 대인 간격과 노출은 `Interpersonal Distance`와 `Occupant Proximity Analysis`로 제공된다.
  - 관련 문서: [Interpersonal Distance](https://www.thunderheadeng.com/docs/2026-1/pathfinder/output/interpersonal-distance/), [Occupant Proximity Analysis](https://www.thunderheadeng.com/docs/2026-1/results/viewing-pathfinder-output/occupant-proximity/)
- 반복 실행은 Monte Carlo variation과 summary 통계로 정리되고, 프로그램 연동은 cumulative JSON 같은 결과 아티팩트 중심으로 설명된다.
  - 관련 문서: [Monte Carlo Results](https://www.thunderheadeng.com/docs/2026-1/pathfinder/output/monte-carlo/), [Cumulative Output](https://www.thunderheadeng.com/docs/2026-1/pathfinder/output/cumulative/)
- SafeCrowd는 `구역 기반 측정`, `history artifact`, `variation summary`, `post-processing`을 고급 기능의 기본 축으로 삼는 편이 Pathfinder와 가장 잘 맞다.

### 2.3. 시야 저하와 환경 연동
- Pathfinder는 시야 저하를 Results의 visibility 계산과, 선택적 FDS/FED 연동으로 다룬다.
  - 관련 문서: [Calculating Occupant Visibility through Smoke](https://www.thunderheadeng.com/docs/2026-1/results/viewing-pathfinder-output/visibility/), [Calculating Fractional Effective Dose (FED)](https://www.thunderheadeng.com/docs/2026-1/results/viewing-pathfinder-output/fed/), [Coupling with PyroSim (FDS) Simulations](https://www.thunderheadeng.com/docs/2026-1/pathfinder/advanced/coupling-fds/)
- 여기서 바로 가져올 수 있는 것은 `시야 저하`, `친숙도`, `유도 신호`, `연기/FED 후처리`의 단계적 결합이다.
- 반면 연기 중 독자 길찾기 알고리즘이나 세부 인지 심리 모델까지 Pathfinder가 정식 기능으로 명세하는 것은 아니다.

### 2.4. 검증 시나리오로 보는 고급 거동
- Pathfinder는 merging, bidirectional flow, stair passing, elevator loading 같은 거동을 제품 기능 자체보다 `검증 시나리오`로 많이 다룬다.
  - 관련 문서: [Behavior Tests](https://www.thunderheadeng.com/docs/2026-1/pathfinder/appendices/verification-validation/behavior/), [Fundamental Diagram for Bidirectional Flow](https://www.thunderheadeng.com/docs/2026-1/pathfinder/appendices/verification-validation/fundamental-diagrams/bidirectional/)
- 따라서 SafeCrowd도 `복도 합류`, `계단 합류`, `양방향 흐름`, `엘리베이터 탑승`을 먼저 검증/비교 케이스로 다루는 편이 자연스럽다.
- 이 항목들을 곧바로 별도 공개 위험 모듈이나 독립 사고 모델로 승격할 필요는 없다.

## 3. 연구 후보

### 3.1. 압사와 압박 판정
- `Compression asphyxia`, `CompressionForce`, `CompressionExposure` 같은 개념은 SafeCrowd에 유용할 수 있지만, Pathfinder 2026.1의 1급 기능 노출 방식과는 거리가 있다.
- Pathfinder에 가까운 현재 범위는 `고밀도 체류`, `근접 노출`, `hotspot`, `interpersonal distance`까지다.
- 따라서 압사 판정은 `후속 연구 확장`으로 두는 편이 맞다.
- 참고 문헌: [Nolan et al., 2021](https://pubmed.ncbi.nlm.nih.gov/32883753/), [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/)

### 3.2. 낙상과 연쇄 전도
- 낙상과 domino effect도 Pathfinder의 직접 기능보다는 SafeCrowd 독자 확장에 가깝다.
- 구현하려면 `사건 트리거`, `동적 장애물화`, `후속 혼잡 영향`까지 별도 설계가 필요하다.
- 따라서 현재는 `즉시 구현 후보`가 아니라 `연구 후보`로만 유지한다.
- 참고 문헌: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/), [Wang et al., 2023](https://pubmed.ncbi.nlm.nih.gov/36894613/)

### 3.3. 역방향 흐름과 교차류의 정밀 지표
- `OrderParameter`, `LaneCount`, `StripeCoherence`는 연구와 검증 관점에서는 유용하지만, Pathfinder가 일반 사용자 기능으로 전면에 내세우는 출력은 아니다.
- Pathfinder의 bidirectional 관련 공식 검증도 `simple lane-forming algorithm` 수준과 결과 비교에 더 가깝다.
- 따라서 이 지표들은 현재 단계에서는 `공개 핵심 KPI`보다 `검증용 분석 지표`로 두는 편이 안전하다.
- 참고 문헌: [Feliciani & Nishinari, 2016](https://pubmed.ncbi.nlm.nih.gov/27739694/), [Mullick et al., 2022](https://pubmed.ncbi.nlm.nih.gov/35679329/)

### 3.4. 추천 규칙과 최적화
- 운영 대안 추천 기능 자체는 제품 범위에 둘 수 있다.
- 다만 `위험 패턴 -> 운영 대안` 매핑 규칙과 자동 탐색 로직은 Pathfinder가 직접 제공하는 기능이 아니므로, 별도 추천 설계 문서에서 다루는 편이 맞다.

## 4. 설계 반영 원칙
- 지금 바로 UML, issue, 구현 계획에 반영해도 되는 축은 `경로 비용`, `trigger response`, `measurement regions`, `history artifact`, `proximity`, `visibility`, `Monte Carlo`, `results post-processing`이다.
- `압사`, `낙상`, `역방향/교차류 정밀 지표`, `추천 최적화`는 별도 설계 문서가 생기기 전까지 구현 계획의 기본 단위로 올리지 않는다.
- 엔진 설계에서는 새 에이전트 단위 컴포넌트를 늘리기보다, 구역/필드/히스토리 아티팩트 중심으로 먼저 정리하는 편이 Pathfinder와 더 가깝다.
- 검증은 기능 설명보다 `benchmark 시나리오와 결과 비교` 중심으로 잡는 편이 적절하다.

## 5. 별도 설계 문서가 필요한 경우
- 아래 항목은 실제 구현 전에 별도 설계 문서가 필요하다.
  - 압사/압박 판정
  - 낙상/연쇄 전도
  - 역방향 흐름 lane/order 지표
  - 교차류 stripe/coherence 지표
  - 추천 규칙/최적화
- 별도 설계 문서에는 최소한 `문제 정의`, `입력 데이터`, `계산 규칙`, `출력 지표`, `검증 근거`, `성능 모드`가 있어야 한다.
