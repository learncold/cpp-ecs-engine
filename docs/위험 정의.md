# SafeCrowd 시뮬레이션: 위험 및 사고 모델링 정의서

## 개요
본 문서는 SafeCrowd ECS 기반 군중 시뮬레이션에서 재현할 위험과 사고 유형을 현실 관찰 및 실험 연구에 맞춰 다시 정의한 문서이다. 목적은 단순히 "위험해 보이는 장면"을 만드는 것이 아니라, 실제 의사결정에 쓸 수 있는 비교 가능한 위험 지표와 시나리오별 차이를 재현하고, 운영 대안 추천의 근거를 제공하는 것이다.

본 개정안은 다음 원칙을 따른다.

- 심리 상태를 직접 추정하지 않고, 관찰 가능한 물리량과 환경 변수로 위험을 설명한다.
- 다만 시야, 공간 친숙도, 표지, 유도 인력 같은 외생 변수는 실제 대피 성능에 영향을 주므로 모델에 포함한다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/), [Xue et al., 2020](https://doi.org/10.1080/23249935.2020.1722281)).
- 위험 징후와 최종 사고를 구분한다. 예를 들어 고밀도, 정체, 압력 집중은 징후이고, 낙상과 압사는 사고 결과다 (주근거: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/), [Nolan et al., 2021](https://pubmed.ncbi.nlm.nih.gov/32883753/)).
- 단일 임계값 하나로 판정하기보다, 강도와 지속시간을 함께 보고 위험도를 계산한다 (보조근거: [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/)).
- 결과는 시나리오 간 비교가 가능하도록 절대값과 상대값을 함께 산출한다.

## 출처 사용 원칙
- **주근거:** 실제 사건 분석, 사람 대상 실증 데이터, 현장 관찰 연구. 위험의 존재와 핵심 현상 정의에 우선 사용한다.
- **보조근거:** 통제 실험, 특정 공간이나 조건에서의 대피 실험. 환경 변수와 경향 설명에 사용한다.
- **구현 보조근거:** 시뮬레이션, 이론 모델, 세포자동자·입자 모델 논문. 알고리즘 설계와 변수 선택의 참고로만 사용한다.

## 성능 구현 원칙
- 본 문서의 변수와 지표는 "현상을 설명하기 위한 개념 모델"이며, 모든 항목을 엔티티 단위 ECS 컴포넌트로 1:1 구현하라는 의미는 아니다.
- 실시간 대규모 시뮬레이션에서는 **핫패스 컴포넌트**와 **분석용 파생 지표**를 분리한다.
- 핫패스 컴포넌트는 매 프레임 이동 계산에 직접 필요한 최소 상태만 유지한다.
- 분석용 지표는 가능하면 공간 분할 그리드, 구역(zone), 출구 단면, 교차 셀 같은 집계 단위에서 계산한다.
- `OrderParameter`, `LaneCount`, `StripeCoherence`, `FlowRate`, `ConflictRate` 같은 값은 기본적으로 엔티티 컴포넌트보다 그리드 캐시 또는 시스템 리소스로 두는 편이 적절하다.
- 매 프레임 전체 재계산이 필요 없는 지표는 저주기 샘플링이나 tick 기반 업데이트를 우선 검토한다.

## 시나리오 설정 및 UI 추상화 원칙
- 사용자는 시나리오를 **직관적인 상위 개념**으로 설정하되, 실제 운영 검토에 필요한 구조·운영 조건은 직접 다룰 수 있어야 한다.
- `StabilityMargin`, `CollisionImpulse`, `FollowingWeight`, `CompressionExposure` 같은 내부 변수는 기본 UI에서 직접 노출하지 않는다.
- 사용자는 `군중 밀집도`, `공간 친숙도`, `시야 상태`, `통제 강도`, `이동 긴급도` 같은 시나리오 레벨 변수와 함께 `출구 개방 여부`, `통로 폭`, `펜스/통제 구역`, `일방통행`, `순차 퇴장`, `장애물 배치` 같은 구조·운영 변수를 설정할 수 있어야 한다.
- 상위 입력값은 다수의 내부 파라미터로 매핑된다. 이때 단일 치환보다 범위 기반 보간(interpolation)과 preset 조합을 우선 사용한다.
- 고급 사용자나 연구 모드가 아니라면, 개별 ECS 컴포넌트 초기값 편집은 숨기거나 개발자용 패널로 분리한다.
- 사용자 친화적 표현으로 `패닉 수준`을 사용할 수는 있지만, 문서의 물리 중심 원칙과의 충돌을 줄이려면 기본 용어는 `이동 긴급도` 또는 `압박 강도`로 두는 편이 더 일관적이다.

### UI 입력과 내부 파라미터 매핑 예시
- **군중 밀집도:** 초기 배치 인원 수, 시작 간격, 국소 밀도 임계치, 출구 유입률에 영향을 준다.
- **공간 친숙도:** `Familiarity`, 길찾기 오차, 표지 의존도, 벽따라가기 가중치에 영향을 준다.
- **시야 상태:** `VisibilityLevel`, `SmokeExposure`, 탐색 시간, 이동 속도 감소 계수에 영향을 준다.
- **통제 강도:** 일방통행 규칙, 유도 신호 강도, 펜스/통제 구역 효과, 역방향 흐름 허용도에 영향을 준다.
- **이동 긴급도:** `DesiredSpeed`, 속도 분산, 접촉 빈도, 병목 유입 압력, 낙상·압력 위험 증가율에 영향을 준다.

### 구현 메모
- 시나리오 로더는 상위 입력값을 받아 내부 파라미터 번들로 변환하는 `ScenarioCalibrationSystem` 또는 `ParameterPresetResolver`를 두는 편이 적절하다.
- 보정 로직은 선형 보간 하나로 고정하지 말고, 공간 유형과 시나리오 유형에 따라 preset curve를 다르게 적용할 수 있어야 한다.
- UI 저장 포맷도 저수준 ECS 값 전체보다 상위 시나리오 파라미터와 preset ID를 저장하는 방식이 유지보수에 유리하다.

### 최소 시나리오 입력 계약
- **공간 구조(`FacilityLayout`):** 방, 복도, 계단, 출구, 교차구역, 장애물, 펜스, 통제 가능 구역을 표현할 수 있어야 하며, 최소한 `구역 ID`, `연결 관계`, `유효 폭`, `경사/계단 여부`, `출입 가능 방향`, `기본 수용 용량`을 포함한다.
- **군중 배치(`PopulationProfile`):** 인원 수, 시작 구역, 기본 목적지, 그룹별 이동 방향, 도착 또는 퇴장 시작 시점, 필요 시 이동 약자 비율 같은 분포 정보를 포함한다.
- **환경 조건(`EnvironmentState`):** 연기/시야, 조명 저하, 표지 가시성, 유도 인력 배치, 기본 개방 출구 목록을 포함한다.
- **운영 계획(`ControlPlan`):** 출구 개방/폐쇄, 일방통행, 구역 접근 제한, 순차 퇴장, 펜스 배치, 임시 장애물 추가 같은 운영 대안을 포함한다.
- **실행 조건(`ExecutionConfig`):** 시뮬레이션 제한시간, 결과 샘플링 주기, 난수 seed, 반복 실행 횟수 같은 실험 설정을 포함한다.
- **입력 계층 원칙:** 사용자가 직접 입력하는 값은 가능한 한 공간·운영·환경 레벨에서 정의하고, `DesiredSpeed`, `FollowingWeight`, `CompressionExposureThreshold` 같은 내부 파라미터는 `ScenarioCalibrationSystem`이 파생한다.

### 운영 이벤트 모델
- **공통 이벤트 포맷:** 모든 시나리오 이벤트는 최소한 `TriggerTime` 또는 `TriggerCondition`, `Target`, `EventType`, `TransitionMode`, `Payload`를 가진다.
- **핵심 이벤트 유형:** `ExitStateChanged`, `ZoneAccessChanged`, `FlowDirectionChanged`, `PhasedReleaseStarted`, `ObstacleInjected`, `GuidanceModeChanged`, `VisibilityChanged` 정도를 공통 이벤트 타입으로 두는 편이 적절하다.
- **즉시 적용과 점진 적용 구분:** 출구 폐쇄나 펜스 설치는 즉시 적용, 연기 확산이나 유도 강도 증가는 시간에 따라 보간되는 점진 적용으로 나누어야 한다.
- **시간 기반과 조건 기반 트리거 병행:** 절대 시간 트리거뿐 아니라 `특정 구역 밀도 초과`, `출구 대기열 길이 초과`, `선행 이벤트 종료` 같은 조건 기반 트리거도 허용해야 운영 시나리오를 자연스럽게 표현할 수 있다.
- **적용 순서 원칙:** 구조/통제 상태 갱신 후 `Navigation Graph`, `GlobalFlowField`, `DynamicCostOverlay` 같은 전역·구역 리소스를 갱신하고, 그 다음 에이전트 지각과 이동을 계산하는 순서를 유지한다.
- **UI 표현 원칙:** 사용자는 타임라인 또는 이벤트 리스트에서 운영 이벤트를 편집하고, 시스템은 이를 내부 필드 갱신과 규칙 변경으로 변환하는 구조가 적절하다.

## 1. 병목, 정체, 탈출 지연 (Bottlenecks, Congestion, Evacuation Delay)

### 1.1. 정의
출구, 계단, 코너, 좁은 통로 같은 기하학적 제약 구간에서 유입량이 처리 용량을 초과해 흐름이 불안정해지고, 자유 흐름에서 정체, `stop-and-go`, 국소 교착 상태로 전이되며 전체 대피 시간이 증가하는 현상이다 (주근거: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/); 보조근거: [Hannun et al., 2022](https://doi.org/10.1016/j.ssci.2021.105696)).

### 1.2. 현실 근거
- 고밀도 군중에서는 흐름이 연속적으로 나빠지는 것에 그치지 않고, 자유 흐름에서 `stop-and-go`와 `turbulent flow`로 급전이할 수 있다 (주근거: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/); 구현 보조근거: [Yu & Johansson, 2007](https://pubmed.ncbi.nlm.nih.gov/17995058/)).
- 병목 폭이 넓어질 때 처리 용량은 항상 선형으로 증가하지 않으며, 동적 lane 형성과 zipper 효과 때문에 구간적으로 증가할 수 있다 (보조근거: [Poojari et al., 2024](https://doi.org/10.2478/sjce-2024-0016)).
- 코너, 굴곡, 계단 같은 복합 지형은 출구 자체가 넓더라도 내부에서 추가 병목을 만들 수 있다 (보조근거: [Hannun et al., 2022](https://doi.org/10.1016/j.ssci.2021.105696)).

### 1.3. 모델링 방안
- **국소 밀도와 유량을 함께 추적:** 반경 기반 `LocalDensity`만이 아니라 단면별 `FlowRate`, `SpecificFlow`, `HeadwayVariance`를 함께 계산한다 (보조근거: [Poojari et al., 2024](https://doi.org/10.2478/sjce-2024-0016)).
- **흐름 상태 전이 도입:** 각 구역을 `Free -> Congested -> StopAndGo -> Gridlock` 상태로 구분하고, 밀도·유량 저하·속도 분산·정체 지속시간을 기준으로 전이시킨다 (주근거: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/)).
- **용어 구분:** 여기서 `Gridlock`은 주로 단일 방향 또는 병목 과포화로 인한 거의 정지 상태를 뜻하고, 역방향 흐름 절의 `Deadlock`은 상반 흐름 집단이 서로 막아 움직이지 못하는 상태를 뜻한다.
- **병목 처리 용량을 기하학적으로 계산:** 출구 폭과 통로 폭에 따라 lane 수를 추정하고, 처리 용량이 선형이 아니라 구간별로 증가하도록 설정한다 (보조근거: [Poojari et al., 2024](https://doi.org/10.2478/sjce-2024-0016)).
- **코너/계단 보정:** 회전 각도, 경사, 유효 폭 감소를 반영해 병목 계수를 추가한다 (보조근거: [Hannun et al., 2022](https://doi.org/10.1016/j.ssci.2021.105696)).
- **탈출 지연과 탈출 실패 분리:** 총 대피 시간, 90/95 퍼센타일 대피 시간, 시나리오 종료 시 미도달 인원, 설정 제한시간 초과 인원을 별도로 기록한다.

### 1.4. ECS 컴포넌트 및 시스템
- **핫패스 Components:** `Position`, `Velocity`, `DesiredSpeed`, `Goal`, `GeometryConstraint`
- **그리드/구역 캐시:** `LocalDensityField`, `FlowRateField`, `FlowStateField`
- **Systems:** `CrowdDensitySystem`, `FlowMeasurementSystem`, `CongestionStateSystem`, `MovementSystem`
- **구현 메모:** `LocalDensity`, `FlowRate`, `FlowState`는 엔티티별 컴포넌트보다 구역 또는 셀 단위 캐시에 두는 편이 대규모 시뮬레이션에 유리하다.

## 2. 압력 집중 및 압사 위험 (Compression Load & Crush Asphyxia)

### 2.1. 정의
극단적 고밀도 상태에서 사람-사람, 사람-벽, 사람-장애물 접촉이 지속되며 흉부와 복부에 압축 하중이 누적되고, 이 하중이 충분히 크거나 오래 지속되어 호흡이 제한되는 위험 상태 및 사고 결과이다 (주근거: [Nolan et al., 2021](https://pubmed.ncbi.nlm.nih.gov/32883753/); 보조근거: [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/)).

### 2.2. 현실 근거
- 실제 군중 참사 분석에서는 다수 사망의 직접 원인이 `compression asphyxia`였다 (주근거: [Nolan et al., 2021](https://pubmed.ncbi.nlm.nih.gov/32883753/)).
- 흉복부 압박 실험은 총 하중뿐 아니라 압박 위치와 지속 시간이 호흡 부전 위험에 중요함을 보여준다 (보조근거: [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/)).
- 따라서 압사는 "순간 최대 힘" 하나보다 `접촉 하중 x 지속시간`으로 보는 편이 현실에 가깝다 (보조근거: [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/)).

### 2.3. 모델링 방안
- **접촉 하중 표현:** 기본 구현에서는 `CompressionForce` 단일 값으로 계산하고, 고정밀 분석이 필요할 때만 `ChestLoad`, `AbdominalLoad` 같은 세분 항목을 선택적으로 확장한다 (보조근거: [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/)).
- **지속시간 누적:** 압축 하중이 위험 구간에 들어간 시간을 누적해 `CompressionExposure`로 관리한다 (보조근거: [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/)).
- **벽 끼임 보정:** 동일한 하중이라도 에이전트 뒤가 막혀 있거나 벽면에 끼여 있으면 위험 가중치를 높인다. 이때 `WallPinningState`를 별도 보관하기보다 접촉 계산 단계에서 가중치로 바로 반영해도 된다 (주근거: [Nolan et al., 2021](https://pubmed.ncbi.nlm.nih.gov/32883753/)).
- **단일 임계값 대신 위험 구간 사용:** `주의`, `고위험`, `치명` 같은 다단계 위험 구간을 두고, 하중과 지속시간 조합으로 판정한다 (보조근거: [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/)).
- **결과와 징후 분리:** 압사 상태 이전에도 압력 집중 구역, 평균 접촉 하중, 최대 노출시간을 별도 시각화한다.
- **용어 메모:** `CompressionCriticalState`는 시뮬레이션상 치명적 압박 조건 충족 상태를 뜻하는 내부 판정이며, 실제 의학적 사망 판정과 동일한 의미로 사용하지 않는다.

### 2.4. ECS 컴포넌트 및 시스템
- **핫패스 Components:** `Mass`, `Collider`, `CompressionForce`, `CompressionExposure`
- **파생 상태/이벤트:** `CompressionCriticalState`
- **선택적 정밀 모드:** `ChestLoad`, `AbdominalLoad`
- **Systems:** `ContactForceSystem`, `CompressionLoadSystem`, `AsphyxiationRiskSystem`
- **구현 메모:** 기본 프로토타입은 `CompressionForce + wall pinning weight` 조합으로 시작하고, 의학적 세분화가 꼭 필요한 경우에만 분리 값을 추가하는 것이 성능상 안전하다.

## 3. 낙상 및 연쇄 전도 (Falls & Domino Effect)

### 3.1. 정의
밀침, 접촉 충격, 급격한 속도 변화, 발 디딤 공간 부족으로 인해 보행 안정성이 무너져 넘어지고, 넘어진 인원이 후속 흐름을 방해해 추가 낙상과 짓밟힘 위험을 유발하는 현상이다 (주근거: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/); 구현 보조근거: [Wang et al., 2023](https://pubmed.ncbi.nlm.nih.gov/36894613/)).

### 3.2. 현실 근거
- 고밀도 군중에서는 난류적 변위와 급격한 압력 방출이 사람을 밀어 넘어뜨릴 수 있다 (주근거: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/); 구현 보조근거: [Yu & Johansson, 2007](https://pubmed.ncbi.nlm.nih.gov/17995058/)).
- 최근 모델링 연구는 단순 충돌보다 `collision impulse`와 `pushing force`가 결합될 때 domino effect가 잘 재현됨을 보였다 (구현 보조근거: [Wang et al., 2023](https://pubmed.ncbi.nlm.nih.gov/36894613/)).
- 따라서 낙상은 단순한 "확률 이벤트"보다 안정성 손실의 결과로 계산하는 편이 적절하다 (구현 보조근거: [Wang et al., 2023](https://pubmed.ncbi.nlm.nih.gov/36894613/)).

### 3.3. 모델링 방안
- **안정성 여유도 계산:** 에이전트별 `StabilityMargin`을 두고, 측면 충격량, 속도 변화량, 주변 여유 공간 감소가 누적되면 감소시킨다 (구현 보조근거: [Wang et al., 2023](https://pubmed.ncbi.nlm.nih.gov/36894613/)).
- **충격량 기반 낙상 판정:** 순간 충돌력보다 `CollisionImpulse`와 짧은 시간 내 반복 충격 횟수를 함께 본다 (구현 보조근거: [Wang et al., 2023](https://pubmed.ncbi.nlm.nih.gov/36894613/)).
- **낙상 후 장애물화:** 낙상한 에이전트는 낮은 높이의 동적 장애물로 전환되며, 뒤따르는 에이전트의 회피 공간을 줄인다 (주근거: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/)).
- **연쇄 전도 조건:** 후속 에이전트가 낮은 여유 공간과 높은 충격량 조건을 동시에 만족할 때 추가 낙상 위험을 크게 높인다 (구현 보조근거: [Wang et al., 2023](https://pubmed.ncbi.nlm.nih.gov/36894613/)).
- **후속 위험 연결:** 낙상은 곧바로 압력 집중과 탈출 지연 위험을 증가시키므로 인접 셀의 압력 위험도와 연동한다 (주근거: [Helbing et al., 2007](https://pubmed.ncbi.nlm.nih.gov/17500963/)).

### 3.4. ECS 컴포넌트 및 시스템
- **핫패스 Components:** `StabilityMargin`, `CollisionImpulse`, `FreeSpace`, `FallenState`
- **파생 장애물 표현:** `GroundObstacle`
- **Systems:** `StabilitySystem`, `FallDetectionSystem`, `DynamicObstacleSystem`, `SecondaryRiskSystem`
- **구현 메모:** `GroundObstacle`는 별도 엔티티나 충돌 레이어 전환으로 표현할 수 있으며, 모든 낙상 에이전트에 즉시 추가 컴포넌트를 붙이는 방식만이 정답은 아니다.

## 4. 역방향 흐름 (Counter-flow)

### 4.1. 정의
같은 공간 안에서 서로 반대 방향의 보행 집단이 동시에 이동하면서 횡방향 회피, lane 재구성, 속도 저하가 반복되고, 밀도가 올라가면 lane 붕괴와 국소 교착으로 이어지는 현상이다 (주근거: [Feliciani & Nishinari, 2016](https://pubmed.ncbi.nlm.nih.gov/27739694/); 구현 보조근거: [Weng et al., 2006](https://doi.org/10.1103/PhysRevE.74.036102)).

### 4.2. 현실 근거
- 역방향 보행에서는 자발적인 lane formation이 반복적으로 관찰되며, 이는 저밀도에서는 효율을 높이는 자기조직화 현상이다 (주근거: [Feliciani & Nishinari, 2016](https://pubmed.ncbi.nlm.nih.gov/27739694/); 구현 보조근거: [Weng et al., 2006](https://doi.org/10.1103/PhysRevE.74.036102)).
- 반대로 밀도가 높아지면 lane 안정성이 약해지고, 평균 속도 급락과 deadlock에 가까운 상태가 발생할 수 있다 (주근거: [Feliciani & Nishinari, 2016](https://pubmed.ncbi.nlm.nih.gov/27739694/)).
- 따라서 역방향 흐름은 단순 충돌 페널티가 아니라 "lane 형성-붕괴"의 관점에서 보는 편이 현실적이다 (주근거: [Feliciani & Nishinari, 2016](https://pubmed.ncbi.nlm.nih.gov/27739694/); 구현 보조근거: [Weng et al., 2006](https://doi.org/10.1103/PhysRevE.74.036102)).

### 4.3. 모델링 방안
- **방향 집단 분리:** 각 에이전트에 `FlowGroup`과 `DesiredDirection`을 부여하고, 반대 방향 집단 간 횡이동과 회피 비용을 계산한다 (주근거: [Feliciani & Nishinari, 2016](https://pubmed.ncbi.nlm.nih.gov/27739694/)).
- **lane 형성 지표 도입:** `OrderParameter`, `LateralVariance`, `LaneCount`를 통해 lane 형성과 붕괴를 측정하되, 이 값들은 복도 또는 셀 단위 집계값으로 계산한다 (주근거: [Feliciani & Nishinari, 2016](https://pubmed.ncbi.nlm.nih.gov/27739694/)).
- **고밀도 deadlock 판정:** 밀도 상승, order parameter 붕괴, 평균 속도 급락이 동시에 나타나면 `DeadlockState`로 판정한다. 이 판정 역시 엔티티별 상태보다 구역 상태로 두는 편이 적절하다 (주근거: [Feliciani & Nishinari, 2016](https://pubmed.ncbi.nlm.nih.gov/27739694/)).
- **마찰 규칙의 제한적 사용:** 벡터 내적 기반 마찰 증가는 보조 항으로만 사용하고, 핵심 상태는 lane 유지 여부로 판정한다 (구현 보조근거: [Weng et al., 2006](https://doi.org/10.1103/PhysRevE.74.036102)).

### 4.4. ECS 컴포넌트 및 시스템
- **핫패스 Components:** `DesiredDirection`, `FlowGroup`
- **그리드/구역 캐시:** `OrderParameterField`, `LaneCountField`, `DeadlockZoneState`
- **Systems:** `CounterFlowInteractionSystem`, `LaneFormationSystem`, `DeadlockDetectionSystem`
- **구현 메모:** `OrderParameter`와 `LaneCount`는 매 프레임 전 엔티티 대상으로 계산하기보다 복도 구간별로 저주기 업데이트하는 편이 현실적이다.

## 5. 교차 흐름 (Cross-flow)

### 5.1. 정의
서로 다른 진행 방향의 보행 집단이 교차각을 이루며 같은 공간을 통과할 때, 충돌 예측과 회피 움직임이 반복되고 줄무늬(stripe) 패턴이나 국소적인 진행 묶음이 형성되면서 통과 효율과 갈등도가 변하는 현상이다 (주근거: [Mullick et al., 2022](https://pubmed.ncbi.nlm.nih.gov/35679329/)).

### 5.2. 현실 근거
- 교차 흐름에서는 bidirectional lane과 다른 자기조직화 패턴이 나타나며, 사람들은 동일 진행 방향의 소규모 묶음처럼 정렬되어 통과한다 (주근거: [Mullick et al., 2022](https://pubmed.ncbi.nlm.nih.gov/35679329/)).
- 이 줄무늬 패턴의 방향과 안정성은 교차각, 유입량, 속도 차이에 영향을 받는다 (주근거: [Mullick et al., 2022](https://pubmed.ncbi.nlm.nih.gov/35679329/)).
- 따라서 교차 흐름은 역방향 흐름과 분리해서, `lane`보다 `conflict`와 `stripe formation` 중심으로 모델링하는 것이 현실적이다 (주근거: [Mullick et al., 2022](https://pubmed.ncbi.nlm.nih.gov/35679329/)).

### 5.3. 모델링 방안
- **교차 갈등도 계산:** 교차각, 상대 속도, 충돌 예측 시간, 최근 회피 횟수를 이용해 `ConflictRate`를 계산하되, 교차 셀 또는 교차 구역 단위로 집계한다 (주근거: [Mullick et al., 2022](https://pubmed.ncbi.nlm.nih.gov/35679329/)).
- **줄무늬 패턴 근사:** 국소적으로 비슷한 진행 방향을 가진 에이전트가 일시적 묶음을 이루도록 허용하고, 묶음 간 간격이 통과 효율을 좌우하게 한다. `StripeCoherence`는 엔티티 컴포넌트보다 구역 분석 지표로 두는 편이 적절하다 (주근거: [Mullick et al., 2022](https://pubmed.ncbi.nlm.nih.gov/35679329/)).
- **교차점 포화 판정:** 교차 셀에서 유입이 유출을 일정 시간 초과하면 `CrossingSaturationState`로 전이시킨다.
- **역방향 규칙과 분리:** 교차류는 `LaneCount`보다 `ConflictRate`, `StripeCoherence`, `CrossingDelay`를 중심 지표로 삼는다 (주근거: [Mullick et al., 2022](https://pubmed.ncbi.nlm.nih.gov/35679329/)).

### 5.4. ECS 컴포넌트 및 시스템
- **핫패스 Components:** `DesiredDirection`
- **그리드/구역 캐시:** `ConflictRateField`, `StripeCoherenceField`, `CrossingDelayField`, `CrossingSaturationState`
- **Systems:** `CrossFlowConflictSystem`, `StripeFormationSystem`, `CrossingSaturationSystem`
- **구현 메모:** `ConflictRate`와 `StripeCoherence`는 매 프레임보다는 일정 tick마다 교차 영역 샘플링으로 계산해도 의사결정 지표로 충분한 경우가 많다.

## 6. 시야 제한, 길찾기 저하, 추종 행동 (Limited Visibility, Wayfinding Degradation, Following)

### 6.1. 정의
연기, 정전, 시각적 장애, 복잡한 구조 때문에 출구 인지와 방향 유지가 어려워져 보행 속도와 경로 선택 능력이 떨어지고, 표지·유도 인력·주변 흐름·벽면 같은 외부 단서를 더 강하게 의존하게 되는 현상이다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/), [Xue et al., 2020](https://doi.org/10.1080/23249935.2020.1722281)).

### 6.2. 현실 근거
- 연기 환경 실험에서는 속도와 의사결정이 단순히 가시성만으로 결정되지 않았고, 공간 친숙도와 절차 숙지도도 큰 영향을 미쳤다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/)).
- 제한 시야 상황에서는 벽을 따라 이동하는 행동이 반복적으로 관찰된다 (보조근거: [Xue et al., 2020](https://doi.org/10.1080/23249935.2020.1722281)).
- 다수 흐름을 따라가는 herd effect는 존재하지만, 항상 목적지 탐색을 완전히 포기하는 것은 아니다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/), [Xue et al., 2020](https://doi.org/10.1080/23249935.2020.1722281)).

### 6.3. 모델링 방안
- **이진 시야 대신 연속 필드:** `VisibilityLevel` 또는 `SmokeDensity`를 구역별 연속값으로 저장한다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/)).
- **속도와 방향 안정성 분리:** 시야가 낮아질수록 `DesiredSpeed`를 감소시키고, 진행 방향 오차와 탐색 시간은 증가시킨다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/)).
- **다중 단서 기반 길찾기:** 출구 기억, 표지 신호, 유도 인력, 주변 흐름 추종, 벽따라가기의 가중치를 합산해 다음 방향을 결정한다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/), [Xue et al., 2020](https://doi.org/10.1080/23249935.2020.1722281)).
- **공간 친숙도 반영:** 처음 보는 공간과 익숙한 공간의 길찾기 실패율을 다르게 둔다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/)).
- **추종 행동은 부분 규칙으로 제한:** 저시야 시 `FollowingWeight`를 높이되, 모든 상황에서 Boids 정렬만 따르도록 단순화하지 않는다 (보조근거: [Porzycki et al., 2018](https://pubmed.ncbi.nlm.nih.gov/30157277/)).

### 6.4. 길찾기 결합 원칙
- **전역 경로는 정적 또는 저주기 갱신 필드로 유지:** 기본 경로 탐색은 출구 기준 `Global Flow Field` 또는 구역 그래프 기반 A* 결과를 사용하고, 에이전트는 이를 기본 진행 방향으로 참조한다.
- **지역 조향은 별도 레이어로 결합:** 실제 이동 방향은 `Global Flow Vector` 위에 `Local Avoidance`, `GuidanceSignal`, `WallFollow`, `FollowingWeight`를 합성해 계산한다.
- **A*의 역할은 계층적으로 제한:** 넓은 공간에서는 매 프레임 개별 A*를 돌리지 않고, 방/복도/계단 단위의 상위 경로만 A*로 정하고 세부 이동은 Flow Field와 로컬 회피가 맡는다.
- **동적 장애물은 우선 로컬 비용장에 반영:** 낙상자, 순간 혼잡, 임시 통제선은 전역 필드를 즉시 재생성하기보다 `Dynamic Cost Overlay` 또는 회피 레이어에 우선 반영한다.
- **전역 재탐색은 이벤트 기반으로 제한:** 출구 폐쇄, 복도 완전 차단, 장시간 deadlock처럼 구조적 변화가 생겼을 때만 영향을 받는 구역의 Flow Field를 재계산한다.
- **시야 필드는 경로를 대체하지 않고 가중치로 작용:** `SmokeDensity`가 높아질수록 전역 벡터 추종 정확도는 낮추고, 표지·벽따라가기·주변 흐름 추종의 비중은 높인다.
- **재탐색은 비동기 또는 tick 기반으로 처리:** 경로 필드 갱신은 매 프레임 동기식으로 하지 않고, 일정 tick마다 또는 이벤트 큐를 통해 부분 갱신하는 구조가 적절하다.

### 6.5. 권장 아키텍처
- **Base Layer:** 정적 구조와 출구 위치를 반영한 `Navigation Graph` 또는 `Global Flow Field`
- **Dynamic Overlay:** 연기, 낙상자, 임시 장애물, 통제 구역을 반영한 `Cost Overlay`
- **Steering Layer:** 충돌 회피, 주변 추종, 벽따라가기, 유도 신호 반영
- **Decision Layer:** 친숙도, 시야 상태, 유도 강도에 따라 어떤 조향 항을 더 신뢰할지 결정
- 이 구조에서는 전역 경로는 드물게 갱신하고, 대부분의 실시간 변화는 `Dynamic Overlay + Steering Layer`에서 흡수한다.

### 6.6. ECS 컴포넌트 및 시스템
- **핫패스 Components:** `Familiarity`, `GuidanceSignal`, `FollowingWeight`, `WallFollowState`, `NavigationIntent`
- **필드/리소스:** `VisibilityField`, `SmokeDensityField`, `GlobalFlowField`, `DynamicCostOverlay`
- **Systems:** `PerceptionSystem`, `WayfindingSystem`, `GuidanceInfluenceSystem`, `LocalSteeringSystem`
- **구현 메모:** 저시야 영향은 매 프레임 에이전트 간 상호작용으로 풀기보다, 구역별 `VisibilityField`와 `GuidanceField`를 참조하는 방식이 더 단순하고 빠르다. `WayfindingSystem`은 전역 필드를 참조해 기본 방향을 만들고, `LocalSteeringSystem`이 낙상자와 주변 군중을 반영해 최종 이동 벡터를 조정하는 구조가 적절하다.

## 7. 위험 결과 시각화, 비교, 추천 근거 지표 (Output, Visualization, Decision Metrics)

위험 요소는 단일 수치가 아니라 시나리오 비교가 가능한 지표 묶음으로 제공되어야 한다.

### 7.1. 실시간 시각화
- **밀도 히트맵:** 구역별 최대 밀도와 고밀도 지속시간 표시
- **압력 히트맵:** 접촉 하중과 압축 노출시간이 높은 구역 표시
- **정체 상태 레이어:** `Congested`, `StopAndGo`, `Gridlock`, `Deadlock` 구간 표시
- **낙상/장애물 레이어:** 낙상 지점과 연쇄 전도 경로 표시
- **교차 갈등도 레이어:** 교차 셀의 `ConflictRate`, `CrossingDelay`, 포화 구간 표시
- **시야 저하 레이어:** 연기 농도, 가시성 저하, 유도 신호 영향 범위 표시

### 7.2. 시나리오 종료 후 지표
- 총 대피 시간
- 50/90/95 퍼센타일 대피 시간
- 시뮬레이션 종료 시 미대피 인원
- 제한시간 초과 인원
- 최대 국소 밀도와 임계밀도 초과 지속시간
- 최대 접촉 하중과 위험 노출시간
- 낙상 인원, 압박 고위험 노출 인원, 압박 임계 상태 진입 인원
- 병목별 처리량과 평균 `headway`
- 역방향 흐름의 `LaneCount`, `OrderParameter`, `Deadlock` 발생 횟수
- 교차 흐름의 `ConflictRate`, `StripeCoherence`, `CrossingDelay`
- 시나리오 간 변화율(delta)

### 7.3. 의사결정용 비교 방식
- 출구 추가 개방, 통로 일방화, 순차 퇴장, 펜스 배치, 통제 구역 설정 같은 운영 대안별로 핵심 지표를 나란히 비교한다.
- 단순 평균값뿐 아니라 최악 구간과 지속시간을 함께 보여준다.
- 위험 구간이 줄어든 대신 다른 구간으로 전이되는지도 표시한다.

### 7.4. 시나리오 비교 및 실험 설계 규칙
- **기준 시나리오 명시:** 모든 대안 비교는 하나의 `BaselineScenario`를 기준으로 수행하고, 어떤 입력값이 달라졌는지 `ChangedVariables`로 함께 기록한다.
- **비교 단위 정규화:** 인원 수나 공간 규모가 다른 시나리오를 비교할 때는 총시간 같은 절대값과 함께 `단위 폭당 유량`, `100명당 낙상 수`, `면적당 고밀도 노출시간` 같은 정규화 지표를 병기한다.
- **재현성 확보:** 같은 시나리오를 비교할 때는 난수 seed를 고정한 단일 실행 결과와 seed를 달리한 반복 실행 요약을 구분해 기록한다.
- **반복 실행 요약:** 확률성 또는 초기 배치 분산이 큰 시나리오는 1회 결과만 제시하지 말고 평균, 중앙값, 90/95 퍼센타일, 최악 실행값을 함께 저장하는 편이 적절하다.
- **종료 조건 통일:** 비교 대상 시나리오는 동일한 제한시간, 동일한 완료 판정 기준, 동일한 실패 처리 규칙을 사용해야 한다.
- **차이 시각화:** 결과 비교는 표 형태뿐 아니라 기준 시나리오 대비 `delta heatmap`, 병목 처리량 변화, 위험 구역 전이 경로를 함께 제시하는 편이 의사결정에 유리하다.

### 7.5. 운영 대안 추천 규칙
- **초기 버전은 규칙 기반 추천으로 시작:** 추천 엔진은 최적화 솔버보다 먼저, 위험 패턴과 운영 대안 매핑 규칙을 사용해 후보를 제안하는 편이 구현과 설명 가능성 면에서 적절하다.
- **추천은 위험-대안 매핑으로 생성:** 예를 들어 병목 집중에는 `출구 추가 개방`, `순차 퇴장`, `유입 분산`을, 역방향 갈등에는 `일방통행`, `동선 분리`, `통제 구역 확장`을 우선 후보로 둔다.
- **각 추천안은 근거를 함께 제시:** 추천안에는 목표 hotspot, 악화된 위험 지표, 변경될 운영 변수, 기대 개선 방향을 반드시 붙여야 한다.
- **추천안은 자동 집행하지 않는다:** 추천은 항상 사용자가 검토하고 승인한 뒤 대안 시나리오로 생성하는 흐름을 유지한다.
- **trade-off를 함께 표시:** 특정 구간 위험은 줄이지만 다른 구간 지연을 늘릴 수 있으므로, 추천안마다 잠재적 부작용이나 재검토 포인트를 함께 기록한다.

## 8. 보정 원칙 및 한계

- 본 모델은 군중의 집단 거동과 공간 위험을 비교하기 위한 시뮬레이션 모델이며, 개인 생존 확률을 의학적으로 예측하는 도구가 아니다.
- 문서 내 `CompressionCriticalState` 같은 내부 상태명은 의학적 확정 판정이 아니라, 운영 대안 비교를 위한 시뮬레이션 위험 상태를 뜻한다.
- 압사와 낙상 판정 임계값은 보편 상수로 두지 않고, 문헌값과 시범 실험을 기반으로 보정 가능한 파라미터로 둔다 (보조근거: [Motomura et al., 2020](https://pubmed.ncbi.nlm.nih.gov/31434839/); 구현 보조근거: [Wang et al., 2023](https://pubmed.ncbi.nlm.nih.gov/36894613/)).
- 실제 현장에 적용할 때는 시설 유형, 인구 구성, 통로 재질, 경사, 가시성 조건에 맞는 캘리브레이션이 필요하다.
- 대규모 실시간 시뮬레이션에서는 현상 충실도보다 실행 성능이 우선일 수 있으므로, 초기 버전은 최소 상태 표현으로 구현하고 정밀 지표는 선택적 분석 모드로 분리하는 것이 바람직하다.
