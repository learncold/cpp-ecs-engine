# SafeCrowd 시스템 아키텍처 설계서

## 1. 목적
SafeCrowd 프로젝트의 코드 구조와 빌드 구조를 정리한 문서

이번 설계의 기준은 다음 두 가지다.

- 현재 프로젝트는 Qt 기반 애플리케이션으로 구현한다.
- 프로젝트 종료 후에는 **ECS 엔진 자체를 분리해서 별도 졸업 프로젝트 자산으로 가져갈 수 있어야 한다.**

따라서 구조는 복잡하게 쪼개지 않고, 다음 3계층만 유지한다.

- `engine`
- `domain`
- `application`

---

## 2. 핵심 원칙

### 2.1. 계층 원칙
- `engine`은 범용 ECS 엔진이다.
- `domain`은 SafeCrowd 문제 영역이다.
- `application`은 Qt UI와 실행 앱이다.

### 2.2. 분리 원칙
- `engine`은 SafeCrowd를 몰라야 한다.
- `domain`은 Qt를 몰라야 한다.
- `application`은 엔진과 도메인을 조립하는 역할만 한다.

### 2.3. 단순화 원칙
- 처음부터 세부 폴더를 많이 만들지 않는다.
- 일단 3계층만 물리적으로 분리한다.
- 계층 안에서 파일 수가 많아질 때만 하위 폴더를 추가한다.

## 4. 계층별 책임

### 4.1. engine
`engine`은 다른 프로젝트에도 가져다 쓸 수 있는 범용 엔진 코드만 가진다.

포함할 것:
- ECS world, entity, component, system
- update loop, frame scheduling
- renderer, viewport bridge 지원(초기 버전에서는 최소 범위만 제공 가능)
- 공통 유틸리티와 기본 타입

현재 코어 시작 구조:
- `EngineRuntime` : 엔진 상태와 실행 루프를 관리하는 진입점
- `Entity` : `index + generation` 기반 식별자. 재사용된 entity ID의 stale reference를 막기 위한 기본 핸들
- `EngineWorld` : query / resources / commands를 통해 월드 상태에 접근하는 공유 파사드
- `WorldQuery`, `WorldResources`, `WorldCommands` : 컴포넌트 조회·값 접근 / 리소스 접근 / 구조 변경 요청을 분리한 접근면
- `WorldCommands` : 엔티티 생성/삭제와 컴포넌트 추가/제거 같은 구조 변경을 즉시 ECS에 반영하지 않고 command buffer에 기록하며, 실제 반영은 runtime이 phase 경계에서 수행
- `FrameClock` : 고정 timestep 누적과 catch-up step 계산 담당
- `EngineSystem` : `update(world, stepContext)` 형태로 월드와 프레임 컨텍스트를 받는 범용 시스템 인터페이스
- `SystemDescriptor` : phase, order, trigger policy, 의존 순서를 설명하는 실행 메타데이터
- `DeterministicRng` : seed 기반 반복 실행을 위한 결정적 난수 스트림 제공
- `EngineConfig`, `EngineStats`, `EngineStepContext` : 실행 설정과 프레임 메타데이터

초기 엔진 구현 우선순위:
- `EntityRegistry`, `ComponentRegistry`, `PackedComponentStorage`
- `EngineWorld` 파사드와 `WorldQuery` / `WorldResources` / `WorldCommands`
- 고정 timestep 기반 `FrameClock`
- phase 기반 `SystemScheduler`
- 결정적 반복 실행을 위한 `DeterministicRng`

초기 엔진 구현에서 뒤로 미뤄도 되는 항목:
- snapshot export / replay
- event-driven trigger 확장
- 고급 query cache / archetype 최적화
- 도메인 특화 spatial partition / navigation 보조 기능

넣지 말아야 할 것:
- `CrowdAgent`, `FacilityLayout`, `ScenarioEvent`
- 대피, 압력, 낙상, 위험 지표 계산
- Qt 위젯 코드

### 4.2. domain
`domain`은 SafeCrowd 문제 자체를 구현한다.

포함할 것:
- 공간 구조
- 시나리오 데이터
- `FacilityLayout`, `PopulationSpec`, `EnvironmentState`, `ControlPlan`, `ExecutionConfig` 입력 계약
- `FacilityLayout` 안의 `Room`, `Door`, `Connector`, `ControlZone`, `MeasurementRegionSpec` 같은 실행 계약
- baseline과 대안 비교를 위한 `ScenarioVariation`
- 시나리오 보정(`ScenarioCalibrationService`)
- 단일 실행 세션(`SimulationSession`)과 배치 실행(`ScenarioBatchRunner`)
- 시뮬레이션 규칙
- 위험 계산
- run/variation/comparison/cumulative 단위 결과 아티팩트(`RunResult`, `VariationSummary`, `ScenarioComparison`, `CumulativeArtifact`)
- persisted 결과 아티팩트 생성/갱신(`ResultAggregator`)
- 운영 대안 추천(`AlternativeRecommendationService`)

넣지 말아야 할 것:
- `QWidget`, `QMainWindow`, `QOpenGLWidget`
- 메뉴, 버튼, 다이얼로그 같은 UI 코드

### 4.3. application
`application`은 Qt 앱 자체다.

포함할 것:
- `main()`
- 메인 윈도우
- 시나리오 편집 UI
- 실행/정지 버튼
- 결과 표시 화면
- 추천안 검토 및 적용 화면
- domain 호출과 화면 연결

넣지 말아야 할 것:
- ECS 내부 구현
- 위험 계산 공식

## 5. 의존성 규칙

기본 규칙은 다음 한 줄로 정리된다.

`application -> domain -> engine`

즉,
- `domain`은 `engine`을 사용한다.
- `application`은 `domain`을 사용한다.

### 예외
Qt viewport가 엔진 렌더러와 직접 연결되어야 하면 `application -> engine` 의존은 허용할 수 있다.

하지만 다음은 금지한다.

- `engine -> domain`
- `engine -> application`
- `domain -> application`
- `domain -> Qt`

이 규칙만 지켜도 나중에 `engine`만 떼어내기 훨씬 쉬워진다.

## 7. 빌드/의존성 정리 원칙

### 7.1. 소스 루트 원칙
- 모든 C++ 소스와 헤더는 `src/` 아래에 둔다.
- 각 계층의 헤더와 구현 파일은 기본적으로 `src/<layer>/` 바로 아래에 둔다.
- 따라서 include는 `#include "engine/..."`, `#include "domain/..."`, `#include "application/..."` 형태를 사용한다.

### 7.2. CMake 원칙
- `CMakeLists.txt`의 타깃 소스 경로는 항상 `src/...`를 기준으로 적는다.
- include directory는 `src/`를 공통 공개 루트로 잡는다.
- 계층이 늘어나도 당분간 루트 `CMakeLists.txt` 하나에서 관리하고, 타깃 수와 파일 수가 커질 때만 하위 `CMakeLists.txt`를 추가한다.

### 7.3. 외부 의존성 원칙
- 기본 정책은 `vcpkg.json`으로 의존성을 관리하는 것이다.
- `external/`은 직접 vendoring 해야 하거나 로컬 패치가 필요한 라이브러리만 둔다.
- `external/`에 넣은 라이브러리는 실제로 연결할 때만 CMake에 등록한다.
- 사용하지 않는 vendored 코드는 구조를 흐리므로 장기간 방치하지 않는다.
- 현재 앱 의존성인 `qtbase`는 기본 feature를 끄고 `widgets`만 요청해서, 사용하지 않는 SQL/PostgreSQL 경로까지 로컬 configure에 끌어오지 않는다.

### 7.4. 빌드 디렉터리 원칙
- generator가 바뀌어도 캐시 충돌이 나지 않도록 generator별 하위 빌드 디렉터리를 사용한다.
- 현재 Visual Studio 프리셋은 `build/vs2022/...` 아래에 생성한다.

### 7.5. import stack 로컬 빌드 원칙
- import 관련 오픈소스 스택 연결은 `safecrowd_domain`에서만 켜고, `ecs_engine`에는 연결하지 않는다.
- 기본값은 `SAFECROWD_ENABLE_IMPORT_STACK=OFF`로 두어 CI와 Sprint 1 기본 경로를 그대로 유지한다.
- 로컬 개발 환경에서 import 스택을 켤 때는 `SAFECROWD_IMPORT_STACK_ROOT`를 공통 prefix로 쓰고, 필요하면 `SAFECROWD_LIBDXFRW_ROOT`, `SAFECROWD_IFCOPENSHELL_ROOT`, `SAFECROWD_CLIPPER2_ROOT`, `SAFECROWD_BOOST_ROOT`, `SAFECROWD_RECAST_ROOT`로 개별 경로를 덮어쓴다.
- configure 단계 smoke check는 `libdxfrw`, `IfcOpenShell` parser 헤더, `Clipper2`, `Boost.Geometry`, `Recast`, `Detour`의 최소 include/lib 연결을 확인한다.
- 실제 DXF/IFC adapter 소스가 들어오면 같은 domain 경계 안에서 이 스위치를 재사용하고, 외부 SDK 헤더는 계속 `application`과 `engine` 밖으로 새지 않게 유지한다.
