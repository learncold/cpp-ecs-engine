## 2. 개요

### 2.1 추진 배경

대규모 행사장, 건물, 공공시설에서는 군중 흐름을 사전에 예측하고 병목, 고밀도 체류, 출구 편중 같은 위험을 검토하는 일이 중요하다. 그러나 실제 현장에서는 계획 단계의 판단이 운영자의 경험과 직관에 크게 의존하는 경우가 많다. 발표자료에서 정리한 것처럼 실행 단계보다 계획 단계에 더 많은 노력이 들어가지만, 계획 단계에서 실무자가 사용할 수 있는 기술 지원은 충분하지 않다.

기존 군중 시뮬레이션 도구는 전문가에게 유용하지만, 비전문가가 직접 사용하기에는 다음과 같은 장벽이 있다.

1. 전문 지식과 복잡한 모델 설정이 필요하다.
2. 3D 공간, 보행 모델, 출력 데이터 등 설정 범위가 넓어 사용이 어렵다.
3. 전문가에게 분석을 위임하면 실제 의사결정권자가 시뮬레이션 과정에서 분리된다.
4. 결과 데이터가 원시적이어서 안전관리자가 바로 의사결정으로 연결하기 어렵다.

SafeCrowd는 이러한 문제를 해결하기 위해 군중 관리 실무자가 사용할 수 있는 2D 기반 군중 안전 시뮬레이터를 목표로 한다. 건물 안전 관리자, 행사장 운영자, 시설 관리자처럼 공학적 시뮬레이션 전문 지식이 없는 사용자가 공간 구조를 입력하고, 대피 시나리오를 구성하며, 결과 지표와 추천 대안을 통해 운영 판단을 할 수 있도록 지원한다.

### 2.2 문제 정의

본 프로젝트의 문제 정의는 다음과 같다.

> 군중 관리는 기술적 지원이 필요한 업무이지만, 기존 시뮬레이션 도구는 도메인 지식이 있는 전문가가 아니라면 사용하기 어렵다. 따라서 실무자가 직접 시나리오를 만들고 결과를 해석할 수 있는 단순한 입력 흐름과 의사결정 중심의 결과 표현이 필요하다.

## 3. 목표 및 구현 범위

### 3.1 개발 목표

SafeCrowd의 개발 목표는 다음 네 가지로 정리된다.

1. 비전문가도 사용할 수 있는 Qt 기반 데스크톱 군중 안전 시뮬레이터를 구현한다.
2. 복잡한 3D 모델 대신 층, 구역, 연결부, 장벽, 출구, 계단 중심의 2D 공간 모델을 사용한다.
3. 기준 시나리오와 대안 시나리오를 작성하고, 여러 시나리오를 같은 실행 흐름에서 비교할 수 있게 한다.
4. 대피 시간, 밀도, 병목, 압박 전조, 위험 노출, Cross Flow, 출구 사용률 같은 결과를 직관적인 화면과 지표로 제공한다.

### 3.2 현재 구현 범위

현재 저장소 기준으로 구현된 범위는 다음과 같다.

| 구분 | 구현 현황 |
| --- | --- |
| ECS 엔진 | Entity, Component, System, Resource, CommandBuffer, Scheduler, 고정 timestep, 결정적 난수 seed |
| 공간 입력 | DXF 기반 import, RawImportModel, GeometryNormalizer, FacilityLayoutBuilder, ImportValidationService |
| 레이아웃 검토/편집 | 검증 이슈 표시, 방/출구/벽/문/장애물/층/계단/U자형 계단 편집, 스냅, 층별 네비게이션 |
| 시나리오 작성 | 보행자 개인/그룹/동적 source 배치, 출구/연결부 차단, 화재·연기 위험, 경로 안내, 시나리오 복제 및 차이 추적 |
| 시뮬레이션 | ECS 기반 agent 이동, 충돌/겹침 보정, 장벽 회피, 다층/계단 이동, full knowledge 및 local wayfinding, 위험 회피와 route guidance 반영 |
| batch 실행 | 여러 staged 시나리오와 반복 실행을 관리하고 seed를 달리한 실행 결과를 저장 |
| 결과 지표 | 대피 진행률, T50/T90/T95, 밀도, 체류 히트맵, 압박 전조, 병목, Cross Flow, 화재·연기 노출, 출구 사용률, 구역/배치별 완료 |
| 대안 추천 | 병목, 출구 편중, 압박 hotspot, Cross Flow, 제한시간 초과를 근거로 recommended scenario draft 생성 |
| 프로젝트 저장 | 프로젝트 메타데이터, layout review, import artifact, scenario draft, 실행 결과, batch 결과, workspace 상태 저장/복원 |

### 3.3 현재 범위에서 제외한 항목

현재 보고서에서는 다음 항목을 구현 완료 기능으로 쓰지 않는다.

- 3D 기반 정밀 BIM 시뮬레이션
- FDS/PyroSim 연동, FED, 독성 가스량, 상세 연기 농도장 계산
- 실제 군중 데이터 기반 정량 검증 또는 공식 안전 인증 수준의 판정
- 자동 최적화 설계 산출
- 센서 연동, 실시간 관제, 현장 운영 시스템과의 통합

이 항목들은 프로젝트 가치가 없어서 제외한 것이 아니라, 현재 구현 단계에서는 decision-support prototype의 후속 확장 범위로 보는 것이 적절하다.

## 4. 해결 전략

발표자료의 해결 전략은 "범용 시뮬레이션 수준의 복잡성은 군중 관리 실무자에게 필요하지 않다"는 판단에서 출발한다. SafeCrowd는 이를 다음 방식으로 구현하였다.

| 기존 도구의 어려움 | SafeCrowd의 접근 |
| --- | --- |
| 복잡한 3D 공간 모델을 다뤄야 함 | 2D 시설 레이아웃으로 단순화 |
| 사용자가 모든 행동 모델을 직접 설계해야 함 | 대피 시나리오에 필요한 배치, 차단, 위험, 안내 중심으로 입력 제한 |
| 결과 데이터가 원시적임 | 히트맵, 요약 카드, 비교 표, 위험 목록으로 변환 |
| 한 번에 하나의 시나리오만 보기 쉬움 | staged scenario와 batch runner로 여러 대안을 비교 |
| 전문가와 의사결정권자가 분리됨 | 실무자가 UI에서 직접 수정, 실행, 비교, 추천안을 적용 |

사용자 흐름은 다음과 같다.

1. 새 프로젝트 생성 또는 데모 프로젝트 열기
2. DXF 도면 가져오기 또는 내장 demo layout 선택
3. layout 검토 화면에서 오류와 경고 확인
4. 방, 출구, 문, 벽, 장애물, 층, 계단을 필요한 만큼 보정
5. 시나리오 작성 화면에서 보행자, 출구 차단, 화재·연기, 경로 안내 배치
6. 기준 시나리오와 대안 시나리오를 staged 상태로 설정
7. 실행 설정에서 시간 제한, 샘플 간격, 반복 횟수, seed, wayfinding 방식을 지정
8. 여러 시나리오를 실행하고 결과 화면에서 replay와 위험 지표 확인
9. 추천 대안을 새 recommended scenario로 생성하고 다시 실행

## 5. 시스템 구조 설계

SafeCrowd는 유지보수성과 확장성을 위해 `application -> domain -> engine`의 3계층 구조로 설계하였다.

```text
application -> domain -> engine
```

하위 계층은 상위 계층을 알지 못하도록 구성한다. 즉, engine은 SafeCrowd 도메인과 Qt UI를 몰라야 하고, domain은 Qt UI를 몰라야 한다.

| 계층 | 역할 | 대표 구현 |
| --- | --- | --- |
| Engine | 범용 ECS 런타임 | Entity, ComponentRegistry, PackedComponentStorage, EngineRuntime, SystemScheduler, FrameClock |
| Domain | SafeCrowd 핵심 로직 | FacilityLayout2D, ScenarioDraft, ScenarioSimulationRunner, ScenarioRiskMetrics, AlternativeRecommendationService |
| Application | Qt 데스크톱 UI | LayoutReviewWidget, ScenarioAuthoringWidget, ScenarioRunWidget, ScenarioBatchResultWidget |

CMake 타깃도 계층에 맞춰 분리하였다.

| 빌드 타깃 | 담당 계층 | 역할 |
| --- | --- | --- |
| `ecs_engine` | Engine | ECS 런타임, 엔티티/컴포넌트 저장소, 시스템 스케줄러 |
| `safecrowd_domain` | Domain | 도면 변환, 시나리오, 시뮬레이션, 위험 지표, 추천 로직 |
| `safecrowd_app` | Application | Qt 기반 데스크톱 UI와 사용자 워크플로우 |

이 구조 덕분에 시뮬레이션 로직은 Qt 위젯에 직접 묶이지 않고, UI는 domain service와 runner를 통해 기능을 호출한다. 또한 engine은 다른 프로젝트에서도 재사용 가능한 ECS 런타임으로 유지된다.

## 6. 세부 구현 현황

### 6.1 Engine 계층

Engine 계층은 다수의 agent를 일정한 순서와 timestep으로 갱신하기 위한 ECS 기반 런타임이다. Entity는 `index + generation` 구조로 관리하여 삭제된 entity ID가 잘못 재사용되는 문제를 줄였고, Component는 packed storage에 저장하여 같은 타입의 데이터를 효율적으로 조회할 수 있게 하였다.

주요 구성 요소는 다음과 같다.

| 구성 요소 | 내용 |
| --- | --- |
| EntityRegistry | entity 생성, 삭제, generation 관리 |
| ComponentRegistry | component 타입 등록과 signature 관리 |
| PackedComponentStorage | component를 조밀한 배열 형태로 저장 |
| EngineWorld | query, resources, commands 접근면 제공 |
| WorldQuery | component 조합 조회와 hot loop용 `forEach` 제공 |
| CommandBuffer | entity/component 변경을 phase 경계에서 반영 |
| ResourceStore | 시스템 간 공유 runtime resource 관리 |
| SystemScheduler | phase, order, trigger policy 기반 실행 순서 관리 |
| FrameClock | 고정 timestep과 catch-up step 계산 |
| DeterministicRng | 동일 seed 기반 재현 가능한 실행 지원 |

Engine은 SafeCrowd 전용 개념을 포함하지 않는다. 군중, 출구, 위험 지표 같은 개념은 domain 계층에서 component와 system으로 구성된다.

### 6.2 Layout import 및 편집

Layout 변환은 DXF 도면을 시뮬레이션 가능한 `FacilityLayout2D`로 바꾸는 단계이다. 일반 도면은 사람이 보기 위한 선, 도형, 레이어, 블록 정보로 구성되므로, 이를 바로 agent 이동에 사용할 수 없다. SafeCrowd는 다음 파이프라인으로 도면을 처리한다.

1. `DxfImportService`가 DXF 파일을 읽어 raw geometry와 metadata를 수집한다.
2. `RawImportModel`에 원본에 가까운 형태로 저장한다.
3. `GeometryNormalizer`와 `ImportSemanticRules`가 도형을 정규화하고 의미를 분류한다.
4. `FacilityLayoutBuilder`가 층, 구역, 연결부, 출구, 장벽, 계단으로 구성된 layout을 만든다.
5. `ImportValidationService`가 출구 누락, 연결 불가 구역, 좁은 연결부, 잘못된 층 참조, 비수직 층간 연결 등을 검사한다.

Application 계층의 layout 검토 화면은 자동 변환의 한계를 보완하기 위한 편집 기능을 제공한다.

- 방과 출구를 사각형 또는 다각형으로 작성
- 벽, 문, 장애물, 출구 연결부 편집
- 문 폭, door leaf 여부, 연결부 생성
- 층 추가 및 층별 요소 네비게이션
- 계단, 램프, U자형 계단 생성
- 계단 진입 방향 설정
- grid snap과 기존 geometry 기반 snap
- layout 이슈 카드와 approve flow

따라서 SafeCrowd는 도면 import가 완벽하지 않아도 사용자가 검토 화면에서 시뮬레이션 가능한 2D layout으로 보정할 수 있다.

### 6.3 시나리오 작성

시나리오 작성 기능은 `ScenarioDraft`를 중심으로 구성된다. 시나리오는 `Baseline`, `Alternative`, `Recommended` 역할을 가지며, 기준 시나리오를 복제해 대안을 만들고 변경된 항목을 diff key로 추적한다.

현재 시나리오에 포함되는 주요 입력은 다음과 같다.

| 입력 | 내용 |
| --- | --- |
| PopulationSpec | 초기 배치, 그룹 배치, 동적 occupant source |
| EnvironmentState | reduced visibility, familiarity/guidance profile, 화재·연기 hazard |
| ControlPlan | 운영 이벤트, route guidance, evacuation sign, connection block |
| ExecutionConfig | 시간 제한, sample interval, repeat count, base seed, occupant history 기록 여부, wayfinding mode |

시나리오 작성 화면에서는 사용자가 layout 위에 다음 요소를 직접 배치할 수 있다.

- 개별 보행자 또는 그룹 배치
- 일정 간격으로 agent를 생성하는 occupant source
- 출구 또는 연결부 차단
- 화재 및 연기 hazard
- 특정 출구나 연결부로 유도하는 route guidance
- local wayfinding에 사용할 evacuation sign

Route guidance는 영향 반경, 준수율, 최대 우회 허용거리, 적용 시간대를 가진다. 화재·연기 hazard는 위치, 반경, 시작/종료 시간, 심각도를 가지며, 시뮬레이션에서는 속도 저하, 경로 회피, 노출 지표 계산에 사용된다.

### 6.4 시뮬레이션 실행

시뮬레이션은 `ScenarioSimulationRunner`가 domain 입력을 ECS runtime으로 옮기면서 시작된다. runner는 agent seed 생성, occupant source schedule, layout cache, scenario resource, system 등록을 수행한다.

주요 실행 흐름은 다음과 같다.

1. layout과 scenario 입력을 ECS resource로 등록한다.
2. 초기 배치와 occupant source schedule에 따라 agent를 생성한다.
3. agent별 출구 또는 목표 연결부까지의 경로를 계산한다.
4. 매 fixed step마다 이동 방향, 속도, 충돌 회피, 장벽 회피를 계산한다.
5. 출구 차단, 화재·연기, route guidance, 압박 feedback을 반영한다.
6. 다층 layout에서는 계단과 U자형 계단을 통해 floor transition을 처리한다.
7. frame, risk snapshot, result artifacts를 갱신한다.

Wayfinding은 두 가지 모드를 지원한다.

| 모드 | 설명 |
| --- | --- |
| FullKnowledge | 전체 layout과 출구 정보를 알고 가장 적절한 경로를 계획 |
| LocalWayfinding | 표지판과 주변 연결부를 기반으로 부분 지식 방식 이동 |

계단 및 다층 이동은 발표 이후 구현 범위가 확장된 부분이다. 현재 코드는 계단 진입 방향, 수직 이동 중 물리 bucket, U자형 계단 landing, 층 전환 시 display floor와 collision floor 처리를 다룬다.

### 6.5 Batch 실행과 반복 실행

`ScenarioBatchRunner`는 staged 된 여러 시나리오를 같은 실행 화면에서 관리한다. 각 run은 scenario, 현재 frame, risk snapshot, result artifacts, source scenario index, repeat index, seed를 가진다.

반복 실행은 `ExecutionConfig.repeatCount`와 `baseSeed`를 사용한다. repeat count는 domain 상수 기준 최대 20회로 제한되며, 각 반복 실행은 seed를 달리해 별도 run으로 확장된다. 이를 통해 같은 시나리오라도 확률적 요소가 포함될 때 결과 차이를 비교할 수 있다.

### 6.6 위험 지표와 결과 아티팩트

시뮬레이션 결과는 `ScenarioResultArtifacts`에 축적된다. 단순히 agent의 위치만 저장하는 것이 아니라, 의사결정에 필요한 정량 지표와 replay frame을 함께 제공한다.

| 지표 | 구현 내용 |
| --- | --- |
| 대피 진행률 | 시간별 대피 완료 인원, 전체 인원, 완료율 |
| 대피 시간 | T50, T90, T95, 최종 대피 시간, 목표 시간 대비 margin |
| 밀도 | 1.5m grid 기준 인원/m², peak density, high-density duration |
| 체류 히트맵 | agent-second 누적 기반 occupancy heatmap |
| 압박 전조 | 개인공간 침범 pair, pressure score, exposed/critical agent, critical pressure event |
| 병목 | 열린 connection 주변 agent 집중, stall, 평균 속도 저하 |
| Cross Flow | 2.0m grid 기준 비정렬 이동 흐름, 속도 저하, 노출 시간 |
| 위험 노출 | 화재·연기 hazard별 노출 agent-second, peak exposed count, exposure score |
| 출구 사용률 | 출구별 대피 인원, usage ratio, 마지막 통과 시각 |
| 구역/배치 완료 | zone별, placement별 초기 인원과 대피 완료 시간 |

대표 판정 기준은 다음과 같다.

- 밀도 hotspot: 1.5m x 1.5m cell에서 3.55명/m² 이상
- stalled agent: 속도 0.12m/s 이하 또는 경로 진행 정체 0.75초 이상
- 병목: 열린 connection 주변 1.25m 이내에 active agent 3명 이상, 그리고 stalled agent 존재 또는 평균 속도 0.24m/s 이하
- pressure hotspot: 1.5m x 1.5m cell의 occupant interaction pressure score가 1.0 이상
- critical pressure event: critical agent 2명 이상 상태가 1초 이상 지속
- Cross Flow: 2.0m grid에서 4명 이상의 이동 agent, 양쪽 흐름 비율 30% 이상, 평균 속도 0.97m/s 이하

이 기준들은 결과 화면에서 위험 위치를 찾고, 대안 추천의 근거로 사용된다.

### 6.7 대안 추천

`AlternativeRecommendationService`는 완료된 시나리오 결과와 위험 snapshot을 입력으로 받아 운영 대안을 생성한다. 추천은 "정답"을 자동 산출하는 방식이 아니라, 감지된 위험 유형에 맞는 재시뮬레이션 가능한 후보 scenario draft를 만드는 방식이다.

현재 추천에서 다루는 위험과 대안은 다음과 같다.

| 감지 위험 | 추천 방향 |
| --- | --- |
| 차단된 연결부 주변 병목 | 차단 해제 또는 우회 유도 |
| 출구 사용률 편중 | 덜 사용된 출구로 route guidance 추가 |
| 출구/복도 병목 | 병목 지점을 피해 다른 출구 또는 연결부로 안내 |
| 압박 hotspot | 출구 분산 또는 단계적 출발 source로 변경 |
| Cross Flow | 동선 분리 안내 또는 단계적 출발 |
| 제한시간 초과 | 출발 구역 분산, 출구 부하 균등화, staged evacuation |

추천 결과는 `AlternativeRecommendationCandidate`로 표현되며, title, summary, expected improvement, 근거 evidence, recommended scenario를 포함한다. 결과 화면에서는 추천안을 선택해 새 `Recommended` 시나리오로 만들고, 다시 실행하여 개선 여부를 확인할 수 있다.

### 6.8 Application 계층과 UI

Application 계층은 Qt 기반 데스크톱 애플리케이션으로, 다음 화면 흐름을 제공한다.

1. 프로젝트 목록 및 새 프로젝트 생성
2. layout review 및 editing
3. workspace shell과 project navigator
4. scenario authoring canvas와 inspector
5. staged scenario 실행 화면
6. simulation playback canvas
7. batch result, replay, overlay, 비교 그래프, 추천 패널

결과 화면에서는 다음 overlay를 선택할 수 있다.

- Occupancy
- Density
- Pressure
- Hotspots
- Bottlenecks
- Cross Flow
- None

또한 결과 navigation panel에서 병목, hotspot, Cross Flow, 구역별 완료, 그룹별 완료, 추천 대안을 확인할 수 있다. 특정 위험 항목을 선택하면 canvas에서 해당 위치에 focus를 맞추어 사용자가 문제 지점을 빠르게 찾을 수 있게 하였다.

### 6.9 저장 및 재사용

프로젝트 저장 기능은 단순한 프로젝트 이름 저장이 아니라, 실제 작업을 이어서 할 수 있도록 여러 상태를 보존한다.

- 프로젝트 메타데이터
- layout review 상태
- import artifact
- scenario authoring 상태
- connection block, hazard, route guidance
- 실행 중/완료 결과
- replay frame과 result artifacts
- batch result와 현재 result index
- pressure, hazard exposure, Cross Flow 결과
- workspace navigation 상태

이를 통해 사용자는 기존 프로젝트를 다시 열어 layout을 수정하거나, 기존 시나리오를 복제해 새로운 대안을 만들거나, 이전 결과와 새 결과를 비교할 수 있다.

## 7. 개발 결과

발표자료의 시연 흐름에 맞춰 현재 개발 결과를 정리하면 다음과 같다.

### 7.1 프로젝트 선택과 시작

앱은 프로젝트 선택 화면에서 시작한다. 사용자는 새 프로젝트를 만들거나 기존 프로젝트를 열 수 있고, 내장 demo fixture를 통해 즉시 시뮬레이션 흐름을 확인할 수 있다.

### 7.2 Layout 검토 및 편집

DXF import 또는 demo layout을 불러온 뒤 layout 검토 화면에서 시뮬레이션 가능 여부를 확인한다. 문제가 있는 layout은 issue card로 표시되며, 사용자는 방, 출구, 벽, 문, 장애물, 계단, 층 정보를 직접 수정할 수 있다. 수정 후 layout이 검증 가능한 상태가 되면 approve flow를 통해 시나리오 작성 단계로 넘어간다.

### 7.3 시나리오 편집

시나리오 편집 화면에서는 보행자 배치, occupant source, 출구 차단, 화재·연기, 경로 안내를 layout 위에 배치한다. 기준 시나리오를 복제해 대안 시나리오를 만들 수 있으며, 각 시나리오는 staged 상태로 실행 대상에 포함된다.

경로 안내는 영향 범위, 준수율, 최대 우회 거리 같은 값을 가진다. 화재·연기 hazard는 영향 반경과 심각도를 가지며, 실행 중 agent의 속도와 경로 선택, 결과 노출 지표에 반영된다.

### 7.4 실행 설정과 batch 실행

실행 전에는 다음 값을 설정할 수 있다.

- wayfinding mode
- 최대 실행 시간
- 결과 기록 sample interval
- 반복 실행 횟수
- 난수 seed

Staged 된 여러 시나리오는 같은 실행 화면에서 함께 진행된다. 실행 중에는 각 시나리오의 시간 진행률, 대피 완료율, 정체/병목 상태를 확인할 수 있고, 완료 후 결과 화면으로 이동한다.

### 7.5 결과 분석

결과 화면에서는 replay slider로 시뮬레이션 과정을 되감아 볼 수 있고, density, pressure, bottleneck, hotspot, Cross Flow overlay를 선택해 위험 위치를 확인한다.

오른쪽 패널과 하단 비교 영역에서는 다음 정보를 확인할 수 있다.

- 대피 완료율과 최종 대피 시간
- T50/T90/T95
- 출구별 사용률
- 구역별/배치별 완료 시간
- 병목 목록
- hotspot 목록
- hazard exposure
- pressure hotspot과 critical pressure event
- Cross Flow score와 노출 시간
- 기준 시나리오 대비 대안 시나리오 비교

### 7.6 추천 시나리오 생성과 재실행

결과 화면의 추천 패널은 위험 지표를 근거로 추천 후보를 보여준다. 사용자는 추천안을 선택해 새 recommended scenario를 생성할 수 있다. 생성된 시나리오는 다시 authoring 화면과 run 화면으로 연결되며, 사용자는 추천안이 실제로 개선 효과를 내는지 재시뮬레이션으로 확인한다.

이 흐름은 발표자료에서 강조한 "한 번에 여러 시나리오를 실행 및 비교"와 "직관적인 결과 제시"를 실제 UI 워크플로우로 옮긴 결과이다.

## 8. 검증 현황

프로젝트는 CTest 기반 테스트를 통해 engine, domain, 일부 application helper를 검증한다. 현재 테스트 파일은 ECS core, scheduler, resource, command buffer, DXF import, facility layout builder, scenario authoring, simulation runner, simulation systems, batch runner, recommendation service, persistence codec 등을 다룬다.

대표 검증 범위는 다음과 같다.

- ECS entity/component lifecycle과 stale entity 방지
- frame clock과 fixed timestep 처리
- system scheduler의 phase/order/interval 실행
- DXF import와 layout validation
- 다층 계단 이동과 U자형 계단 floor transition
- occupant source schedule과 repeat seed 확장
- connection block, fire/smoke hazard, route guidance 반영
- local wayfinding과 evacuation sign
- density, pressure, bottleneck, Cross Flow 지표 발행
- result artifacts와 project persistence 저장/복원
- alternative recommendation 후보 생성과 recommended draft 변환

다만 현재 보고서 작성 시점의 검증은 코드 단위 테스트와 로컬 구현 확인 중심이다. 실제 대규모 현장 데이터나 공인 안전 기준과의 정량 검증은 후속 작업으로 남아 있다.

## 9. 기대 효과

### 9.1 군중 안전 위험의 사전 파악

실제 행사나 대피 훈련을 수행하기 전에 가상 환경에서 군중 이동을 시뮬레이션함으로써 병목 구간, 고밀도 구역, 출구 편중, 대피 지연 가능성을 사전에 파악할 수 있다.

### 9.2 비전문가 접근성 향상

복잡한 전문 모델 설정 대신 프로젝트 생성, layout 검토, 시나리오 작성, 실행, 결과 확인의 흐름으로 구성했기 때문에 군중 관리 실무자가 직접 사용할 수 있는 형태에 가깝다.

### 9.3 의사결정 근거 제공

총 대피 시간, T90/T95, 출구별 이용률, 병목 위치, 밀도 hotspot, 압박 전조, Cross Flow, hazard exposure 같은 정량 지표를 제공하여 출구 운영, 동선 안내, 통제 구역 설정에 대한 근거를 제공한다.

### 9.4 운영 대안 비교

기준 시나리오와 대안 시나리오를 같은 결과 화면에서 비교할 수 있어, 특정 조치가 실제로 위험을 줄이는지 재시뮬레이션으로 확인할 수 있다.

### 9.5 반복 실험 비용 절감

대규모 현장 훈련은 인력과 장소, 시간이 많이 필요하지만, 시뮬레이션을 활용하면 여러 조건을 반복적으로 검토할 수 있다. 특히 repeat count와 seed를 활용하면 같은 시나리오에서도 확률적 차이를 점검할 수 있다.

### 9.6 확장 가능한 구조 확보

Engine, Domain, Application이 분리되어 있어 향후 더 정교한 위험 모델, 다른 UI, 추가 import 형식, 외부 분석 도구와의 연동을 비교적 명확한 경계 안에서 확장할 수 있다.

## 10. 현재 한계와 향후 보완

### 10.1 입력 데이터 확장

현재 주요 입력 경로는 DXF와 내장 demo layout이다. DWG, IFC, BIM, FDS 결과와의 직접 연동은 구현 완료 범위가 아니다. 도면 표현이 불완전하거나 의미 분류가 애매한 경우에는 사용자가 layout review 화면에서 직접 보정해야 한다.

### 10.2 화재·연기 모델 한계

현재 화재·연기 기능은 위치, 반경, 심각도, 시작/종료 시간 기반의 경량 hazard model이다. agent의 속도 저하, 경로 회피, 노출 요약에는 활용되지만, 실제 연기 확산, 독성 가스량, FED, 가시거리장 계산은 포함하지 않는다.

### 10.3 위험 판정의 검증 한계

밀도, 병목, 압박 전조, Cross Flow 지표는 운영 판단을 돕는 proxy 지표이다. 현재 단계에서는 공인 안전 판정이나 실제 사고 예측을 보장하는 모델로 해석해서는 안 된다. 실제 적용을 위해서는 공개 데이터, 실험 데이터, 전문가 검토를 통한 calibration이 필요하다.

### 10.4 성능과 규모 검증

ECS 구조는 많은 agent를 다루기 위한 기반을 제공하지만, 실제 대규모 시설과 수천~수만 명 규모에서의 성능 검증은 별도 benchmark가 필요하다. 현재 보고서에서는 구현 구조와 기능 동작을 중심으로 설명한다.

### 10.5 UI 완성도와 보고 기능

현재 UI는 프로젝트 생성, 편집, 실행, 결과 확인 흐름을 제공하지만, 제출용 리포트 자동 생성, 시나리오별 결과 export, 사용자 권한/협업 기능은 후속 개선 범위이다.

## 11. 결론

SafeCrowd는 군중 관리 실무자가 직접 사용할 수 있는 군중 안전 시뮬레이터를 목표로, 현재 ECS 기반 엔진, 2D layout 변환과 편집, 시나리오 작성, batch 실행, 위험 지표 시각화, 대안 추천, 프로젝트 저장 기능까지 구현하였다.

프로젝트의 핵심 성과는 복잡한 시뮬레이션 기능을 단순히 많이 넣는 것이 아니라, 사용자가 "공간을 만들고, 시나리오를 바꾸고, 결과를 비교하고, 추천안을 재실행하는" 흐름을 하나의 데스크톱 앱 안에서 수행할 수 있게 만든 점이다.

다만 본 결과물은 아직 공인 안전성 평가 도구가 아니라 구현 중간 단계의 의사결정 지원 프로토타입이다. 향후에는 입력 형식 확장, 화재·연기 모델 정교화, 실제 데이터 기반 검증, 대규모 성능 측정, 결과 보고서 export 기능을 보완하여 실무 활용 가능성을 높일 계획이다.
